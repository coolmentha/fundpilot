## 0. 设计取向

- **薄信号引擎、厚事实账目**：信号引擎本身是无状态纯函数——给当前状态 + 市场指标，出一条建议；每日 14:50 定时任务跑一遍引擎，把当天的信号快照冻进 `SignalLogEntity`（14:50 的市场指标收盘后无法复算，必须当时落库）。事实数据（资金流水、信号快照、操作记录）老老实实落表。
- **单一事实源：账目只存份额，金额永远实时算**（见下方硬性原则）。
- **配置与状态同表**：策略参数和该参数下的运行时状态（前高、四档时间戳等）放同一张表，按 `status` 区分版本。换参数 =
  新建一份，不是改旧的。
- **每日定时生成信号**：每天 14:50 对所有绑定 `EFFECTIVE` 策略的基金跑信号引擎，结果落 `SignalLogEntity`。用户不触发信号计算，只回应已生成的信号（执行 / override / 主动跳过）；被动忽略（信号已生成但无回应）可追踪。
- **市场指标外部化**：年线/MACD/成交量等本期接入东方财富/天天基金公开接口（ADR-0002），由 `MarketDataFetchJob` 在 14:30/14:
  40/14:50 分批拉取，结果落 `market_indicator_snapshot` 表当缓存；信号生成时从这张表读，不触发外部请求。
- **人工执行、系统记账**：系统出建议，人工拍板；拍板后由系统统一写流水与操作记录，保证两条账对得上。

### 硬性原则：账目只存份额，金额永远实时算

**写入侧（事实账目，下单时写入已知侧，净值确认后回填另一侧，回填后不再改变）**

- `FundTransactionEntity.shares` —— **事实**。买入下单时为 null（已知金额、未知净值，算不出份额），净值确认后回填；卖出下单时已知；回填后永不改变
- `FundTransactionEntity.amount` —— **事实**。买入下单时已知；卖出下单时为 null，净值确认后回填；回填后永不改变
- `FundTransactionEntity.nav` —— 成交净值，净值确认后回填；`amount / shares` 反推也一致
- `FundTransactionEntity.status` —— `PENDING`（已下单等净值确认）/ `CONFIRMED`（净值确认完成）/ `CANCELLED`（撤单）
- `fund_nav_history` —— 基金净值历史表，只读事实数据

> 基金交易的现实时序：14:50 信号 → 15:00 前下单 → 当晚基金公司公布当日净值 → 系统回填另一侧。买入知道金额不知道份额，卖出知道份额不知道金额，是事实尚未发生而非数据缺失。`NavConfirmJob` 每晚净值公布后跑，把 `PENDING` 行回填 `shares`/`amount`/`nav`，转 `CONFIRMED`。

**读取侧（展示/计算，永远实时算，永不落库）**

- 持仓份额 = `Σ shares × direction`（`status = CONFIRMED` 的交易；`PENDING` 算在途，不进持仓）
- 持仓金额 = `持仓份额 × 最近一次确认净值`
- 持仓占比 = `持仓金额 / 总权益持仓金额`
- 浮动盈亏 = `持仓金额 - Σ amount × direction`
- 当前回撤 = `(前高净值 - 最近确认净值) / 前高净值`
- **前高净值（peakNav）= `max(fund_nav_history.accumulated_nav)`**（2026-06-24 修订，ADR-0001，不存字段实时派生）
- **持有期高点（holdingPeriodPeakNav）= 同上 + `WHERE nav_date >= fund.openedAt`**（不存字段实时派生）

**纪律意图（不是事实账目，存在 FundEntity / FundStrategyEntity 上）**

- `FundEntity.plannedTotalAmount`（计划总仓位，基金属性）
- `FundStrategyEntity.tier1Ratio ~ tier4Ratio`（四档基础加仓比例，占 `plannedTotalAmount` 百分比）
- 单品种仓位上限不再存在参数表上，它是全局硬约束规则，落 `HardConstraintConfig` 代码常量

> 准确表述：**事实账目（交易、净值快照）下单时写入已知侧（买入金额 / 卖出份额），净值确认后回填另一侧，回填后永不改变；策略意图（计划仓位、加仓额）存金额；展示用的当前持仓金额永远不落库，永远用
`份额 × 当前净值` 实时算。**

**为什么这么做**

1. 单一事实源：账目只有"买了多少份、什么时候买的"这种客观事实；"现在值多少钱"是派生信息，跟着净值走。两边永远不会矛盾。
2. 净值变动零成本：今天净值涨 3%，不需要写任何数据、不跑任何 job，查询时自然反映。如果维护金额，得每天跑 job 刷所有持仓的
   `amount` 字段，跑挂了账目就错。
3. 历史可复现：任何一天的市值都能算（那天的份额 × 那天的净值）。如果存金额，历史就锁死成"那天跑 job 时算出的值"，跑漏一天丢一天。
4. 净值修正能自然吸收：基金公司偶尔修正前日净值（分红除权、错账修正等）。存金额的话所有历史展示都得重算；份额制下只要修正净值表那一行，所有展示自动正确。

### 硬性原则：归档 = 软删除，与持仓状态正交

归档的真实场景是"我不想在这个平台看见这只基金了"（上错了、不想再看了、没买过也不打算买）。这不是持仓状态的一部分，直接用
`AbstractEntity` 现有的 `@SQLRestriction("deleted_date IS NULL")` + `@SQLDelete` 表达（2026-06-24 修订，ADR-0004 取代原 `@SoftDelete` 方案，因 Hibernate 7 硬性拒绝 `@SoftDelete` 实体的 LAZY 关联）：

- **归档动作** = 执行软删除（`deletedDate = now`，由 `@SQLDelete` 把 JPA `remove()` 重定向为 `UPDATE ... SET deleted_date = now()`）
- **取消归档** = 恢复（`deletedDate = null`），状态回到归档前的样子
- 默认查询天然过滤已软删记录（Hibernate `@SQLRestriction` 自动给所有查询追加 `WHERE deleted_date IS NULL`）
- 不引入 `visibility` / `archived` 这种冗余字段

**关键边界：**

1. **任意 `FundStatus` 都允许软删**，包括 `HOLDING`。不卡校验、不强制填原因——用户可以强制归档仍有持仓的基金。
2. **归档时关联数据一起软删**：`FundTransactionEntity`、`SignalLogEntity`、`FundStrategyEntity`、`FundNavHistoryEntity`
   中该基金的所有关联记录同步软删。恢复归档时一并恢复。理由：归档是"我不想看见这只基金了"
   ，关联数据是这只基金的衍生信息，跟着基金一起隐藏最自然，不留半残数据。
3. **软删 ≠ 物理删除**：数据物理保留，只是从视图隐藏。物理清理过期归档数据另起独立任务，不混进归档动作。

## 1. 分层结构

沿用现有目录约定，不引入新分层：

```
com.fundpilot.backend
├── common/                    # 已有：AbstractEntity / ApiResponse / EnumValue
├── exception/                 # 新增：GlobalExceptionHandler + 业务异常
├── fund/
│   ├── entity/                # 扩展：FundStrategyEntity；新增：SignalLogEntity；改造：FundFlowEntity → FundTransactionEntity
│   ├── enums/                 # 新增：FundCategory / StrategyParamStatus / SignalType / NavSource；改造：FundStatus / FundFlowSource → FundTransactionSource / FundFlowStatus → FundTransactionStatus
│   ├── repository/            # 新增：4 个 JpaRepository
│   ├── dto/                   # 新增：策略配置 / 信号视图 / 操作确认
│   ├── service/
│   │   ├── DisciplineStrategyService   # 信号引擎：每日生成 + 操作确认
│   │   ├── StrategyConfigService       # 参数 CRUD + 状态机
│   │   ├── FundPositionService         # 事实仓位聚合（读 FundTransaction，仅 CONFIRMED）
│   │   ├── SignalGenerationJob         # 每日 14:50 定时任务入口

> 不引入 Hexagonal / DDD 风格的 domain/application/infrastructure 分层。项目体量还撑不起，强行做反而干扰阅读。

## 2. 实体设计

### 2.1 `FundEntity`（基金产品）

只描述"是哪只基金" + 跟策略无关的**基金属性和市场事实 / 持有事实**，不存仓位事实。

| 字段                   | 类型                         | 说明                                                                      |
|----------------------|----------------------------|-------------------------------------------------------------------------|
| 继承自 AbstractEntity   |                            | id / version / 审计 / 软删                                                  |
| fundCode             | String                     |                                                                         |
| fundName             | String                     |                                                                         |
| investmentTarget     | InvestmentTarget enum      | 已有                                                                      |
| operationMode        | OperationMode enum         | 已有                                                                      |
| investmentPhilosophy | InvestmentPhilosophy enum  | 已有                                                                      |
| fundCategory         | FundCategory enum          | 宽基 / 行业 / 主动 / 混合（基金本身属性，用来查默认档位和硬约束上限；跟参数版本无关）                         |
| fundSubType          | FundSubType enum, nullable | 数据源维度分类：ETF / INDEX / INDEX_ENHANCED / ACTIVE（决定行情数据源和逻辑止损判定路径）         |
| benchmarkIndexCode   | String, nullable           | 跟踪/基准指数代码（如 `000300.SH`），指数类基金自动识别填入，主动类基金留空                            |
| status               | **FundStatus**（重做）         | 未建仓 / 持仓中 / 已清仓                                                         |
| plannedTotalAmount   | BigDecimal, nullable       | 计划总仓位（纪律意图）；建仓时（PENDING_HOLDING → HOLDING）必须填，HOLDING/CLEARED 状态下非 null |
| openedAt             | Instant, nullable          | 首次建仓时间（持有事实，跟策略版本无关）；`CLEARED → PENDING_HOLDING` 时清空                    |

**移除（2026-06-24 修订，ADR-0001）**：`peakNav` 和 `holdingPeriodPeakNav` 字段。它们是 `fund_nav_history.accumulated_nav` 的
max 派生值，落库存在净值修正/补录/job 异常三种失真风险。改为实时派生查询，`FundPositionService` 或 `MarketFactService` 提供
`getPeakNav(fundId)` / `getHoldingPeriodPeakNav(fundId)` 方法。

**移除**：现有的 `amount` 和 `shares` 字段。事实仓位由 `FundTransactionEntity` 聚合得出，不在产品实体上冗余。

> `plannedTotalAmount` / `openedAt` 都从 `FundStrategyEntity` 搬过来——它们跟策略参数版本无关，是基金本身的属性或持有事实。换参数版本不影响这些值。

> **`FundStatus` 变更不走日志表**：变更不频繁（一只基金一辈子也就 PENDING_HOLDING → HOLDING → CLEARED → 偶尔回
> PENDING_HOLDING），没有审计/复盘需求，当前状态字段足够。如果以后真需要审计，再补日志表不迟。

### 2.2 `FundStrategyEntity`（策略参数 + 该版本下的运行时状态）

一份记录 = 一份策略快照。改参数 = 新建一份 `PENDING_CALIBRATION`，不是改旧的。激活新版时把旧 `EFFECTIVE` 回退到 `CALIBRATED`（不再用独立的 `RETIRED` 状态）。

只放跟**策略参数版本相关**的字段。基金属性（`plannedTotalAmount`、`openedAt`、`fundCategory`、`fundSubType`、
`benchmarkIndexCode`）和全局硬约束（`singlePositionLimit`）都不在这里——`fundCategory` 现挂在 `FundEntity`
上（基金本身属性，跟参数版本无关）。`peakNav` / `holdingPeriodPeakNav` **既不在这里也不在 `FundEntity`**，由
`fund_nav_history` 实时派生（ADR-0001）。

| 字段                            | 类型                        | 说明                                                                       |
|-------------------------------|---------------------------|--------------------------------------------------------------------------|
| 继承自 AbstractEntity            |                           |                                                                          |
| fund                          | ManyToOne FundEntity LAZY |                                                                          |
| status                        | StrategyParamStatus enum  | 待校准 / 已校准 / 已生效（3 态，无 `RETIRED`，见 3.2）                                   |
| tier1Drawdown ~ tier4Drawdown | BigDecimal                | 四档回撤阈值（百分比，如 0.08 = 8%）                                                  |
| tier1Ratio ~ tier4Ratio       | BigDecimal                | 四档加仓比例（占 plannedTotalAmount 的百分比，如 0.15 = 15%）；四档之和 ≤ 90%                |
| weeklyCoolDownThreshold       | BigDecimal                | 单周跌幅冷静阈值（百分比，如 0.10 = 10%），用于强制冷静                                        |
| stopLossPullbackPercent       | BigDecimal                | 移动止盈回落幅度（百分比，如 0.08 = 8%）                                                |
| tier1AddedAt ~ tier4AddedAt   | Instant, nullable         | 方案B 四档**实际执行**时间戳，null = 未加。**只在用户通过 POST /operations 确认执行时写入**，信号生成只读不写 |

> **不在主表存 `activatedAt` / `deactivatedAt`**：走激活表 `FundStrategyActivationEntity`（见 2.7），主表只保留当前状态 `status`。
> **不在主表存 `openedAt`**：搬到 `FundEntity`（持有事实，跟策略版本无关）。
> **不预留 `userId`**：本期单用户场景，零迁移成本省掉；将来引入多用户时再加 nullable 列即可。

**tier1~4AddedAt 写入时机**（重要）：

- 信号生成（`generateDailySignals`）只读不写 `tierNAddedAt`（例外：反弹清空是状态机自动行为，随信号生成一起写）
- `POST /operations` 确认执行时才写
- 确认的 `SignalLogEntity` 是加仓信号（`signalType=ADD, triggerTier=N`）→ 写 `tierNAddedAt`，依次类推
- 用户不回应 → `tierNAddedAt` 仍是 null → 下次还在该档会再提示

**唯一性约束**：同一 `fund_id` 在未软删的记录里，最多 1 份 `status = EFFECTIVE`。Postgres 部分唯一索引：

```sql
CREATE UNIQUE INDEX uq_fund_strategy_effective
    ON fund_strategy (fund_id) WHERE status = 'EFFECTIVE' AND deleted_date IS NULL;
```

### 2.3 `FundTransactionEntity`（基金交易 · 改名自 `FundFlowEntity`）

原 `FundFlowEntity` 改名为 `FundTransactionEntity`，承载**基金交易事件**——买入、卖出、转入、转出。合并了原 `StrategyOperationEntity` 的策略上下文字段（`signalLog`），手动操作和转账该字段为 null。一条交易对应的"是建仓/加档几/移动止盈/逻辑止损/再平衡"这类策略动作信息**不在交易表上冗余**，要查时通过 `signalLog` 关联读 `SignalLogEntity` 的 `signalType` / `triggerTier` 反推即可。

**字段**

| 字段                    | 类型                                  | 说明                                                                       |
|-----------------------|-------------------------------------|--------------------------------------------------------------------------|
| 继承自 AbstractEntity    |                                     |                                                                          |
| fund                  | ManyToOne FundEntity LAZY           |                                                                          |
| source                | FundTransactionSource enum          | INCREASE / DECREASE / TRANSFER_IN / TRANSFER_OUT / INVEST                  |
| status                | FundTransactionStatus enum          | PENDING / CONFIRMED / CANCELLED                                      |
| shares                | BigDecimal, nullable                | 买入下单时 null（等净值回填）；卖出下单时已知；转账已知；回填后永不改                                      |
| amount                | BigDecimal, nullable                | 买入下单时已知；卖出下单时 null（等净值回填）；转账已知；回填后永不改                                      |
| nav                   | BigDecimal, nullable                | 成交净值，净值确认后回填；下单时 null                                                      |
| confirmTime           | Instant, nullable                   | 净值确认时间（区别于 `createdDate` 下单时间）                                            |
| signalLogEntity       | ManyToOne SignalLogEntity LAZY, nullable | 由信号触发的交易才填；手动操作 / 转账 null                                                |
| relatedFundTransactionEntity | OneToOne FundTransactionEntity LAZY, nullable | 基金转换时指向另一条腿（A 转 OUT + B 转 IN 互指）；其他场景 null                              |

> `FundTransactionEntity` 合并了原 `StrategyOperationEntity` 的职责。一次买入/卖出信号确认 = 写一条 FundTransaction（含 `signalLog`）；`OVERRIDE_SKIP` 不写交易（用户表态跳过 = 无交易记录，与"被动忽略"在数据上不区分）；转账 / 手动操作不带 `signalLog`。策略动作类型（建仓 / 加档几 / 移动止盈 / 逻辑止损 / 再平衡）**不存独立字段**，由 `signalLog.signalType` + `triggerTier` 表达。用户改了建议金额下单（override）**不留痕**——`FundTransaction.amount` 就是用户实际下单金额，系统不关心它和 `SignalLog.suggestedMeasure.value` 是否一致，也不做差值记录。

**交易时序与状态机**

```
PENDING  ── NavConfirmJob 跑完 ──→  CONFIRMED    (买入：回填 shares=amount/nav；卖出：回填 amount=shares×nav)
PENDING  ── 用户撤单 ──→           CANCELLED
```

- 下单那一刻写入已知侧：买入 `amount` 已知、`shares=null`；卖出 `shares` 已知、`amount=null`；`nav=null`、`status=PENDING`
- 当晚基金公司公布当日净值，`NavConfirmJob` 跑：查 `FundNavHistoryEntity` 拿当日净值，回填另一侧 + `nav` + `confirmTime`，转 `CONFIRMED`
- 回填后字段不再改变（符合事实账目"回填后永不改"原则）

**持仓计算**

```
持仓份额 = Σ( shares × direction ) WHERE status = 'CONFIRMED'     # direction: INCREASE/TRANSFER_IN = +1, DECREASE/TRANSFER_OUT = -1
在途订单 = Σ( ... ) WHERE status = 'PENDING'                  # 前端单独展示"已下单待确认"
持仓金额 = 持仓份额 × 当前净值                                       # 实时算，不存
持仓占比 = 持仓金额 / 总权益仓位
```

> 今天净值上涨导致"钱变多了"——不需要任何交易记录，只是 `持仓份额 × 新净值` 自然变大。只有真正发生买卖时才写交易。

**转换（基金 A → 基金 B）**

两条交易通过 `relatedTransaction` 互指：

```
FundTransaction(fund=A, source=TRANSFER_OUT, shares=500, ...)
FundTransaction(fund=B, source=TRANSFER_IN,  shares=500, ...)
  ↑ 两条 relatedTransaction 指向对方
```

撤单时两条一起 CANCELLED。

### 2.4 `SignalLogEntity`（每日信号快照 · 新增）

每日 14:50 定时任务对每只绑定 `EFFECTIVE` 策略的基金跑一遍信号引擎，把当天的信号结果**冻进一张快照表**。信号是当时那一刻的市场切片，14:50 的指标/净值收盘后无法复算，必须当时落库。

| 字段                     | 类型                            | 说明                                                                                                         |
|------------------------|-------------------------------|------------------------------------------------------------------------------------------------------------|
| 继承自 AbstractEntity     |                               |                                                                                                            |
| fund                   | ManyToOne FundEntity LAZY     |                                                                                                            |
| fundStrategyEntity     | ManyToOne LAZY                | 当天生效的参数版本                                                                                                  |
| signalDate             | Instant                       | 信号日期，14:50 那个交易日                                                                                           |
| signalType             | SignalType enum               | NONE / BUILD / ADD / SELL                                                                                  |
| triggerNav             | BigDecimal, nullable          | 触发时净值（signalType ≠ NONE 时）                                                                                 |
| triggerTier            | Integer, nullable             | 触发档位 1~4（ADD 时；移动止盈 SELL 时也用，表示卖第几档）                                                                       |
| coefficient            | BigDecimal, nullable          | 最终系数（ADD 时）                                                                                                |
| suggestedMeasure       | `Measure` @Embedded, nullable | 系统建议量 + 单位：BUILD/ADD 时 `{value, AMOUNT}`（建议金额，买入按金额下单）；SELL 时 `{value, SHARES}`（建议份额，卖出按份额下单）；NONE 时为 null |
| reason                 | String, nullable              | NO_EFFECTIVE_STRATEGY / TIER_HIT / COOLDOWN / WATCH_AND_WAIT / ...                                         |
| warnings               | JSONB / TEXT, nullable        | 强提示列表（强制冷静、破位观望等）                                                                                          |
| hardConstraintBreaches | JSONB / TEXT, nullable        | 命中硬约束被截断的说明                                                                                                |
| generatedAt            | Instant                       | 任务执行时间                                                                                                     |

**唯一性约束**：`(fund_id, signal_date)` 在未软删的记录里唯一——每只基金每个交易日最多一行（含 NONE）。

```sql
CREATE UNIQUE INDEX uq_signal_log_daily
    ON signal_log (fund_id, (signal_date::date)) WHERE deleted_date IS NULL;
```

> 每日一行/基金（含"今天无信号"的 NONE 行），量级和 `fund_nav_history` 同级，兼作"系统每天评估过这只基金"的审计轨迹。
>
> **被动忽略可追踪**：`signalType ≠ NONE` 且无对应 `FundTransactionEntity` 的 SignalLog 行 = 系统提示了但用户没回应（含主动 SKIP 和被动忽略，两者在数据上不区分）。`LEFT JOIN fund_transaction ON signal_log.id = fund_transaction.signal_log_id WHERE transaction.id IS NULL AND signal_type <> 'NONE'` 直接列出。
>
> **`suggestedMeasure` 是带单位的量值**：值对象 `Measure { BigDecimal value; MeasureUnit measureUnit }`，用 `@Embedded`
> 映射成两列 `suggested_value` + `suggested_measure_unit`。由 `measureUnit` 决定 `value` 的语义，`BUILD`/`ADD` 时
`measureUnit=AMOUNT`、`value` 是建议金额（买入按金额下单），`SELL` 时 `measureUnit=SHARES`、`value`
> 是建议份额（卖出按份额下单）。另一侧（买入的份额、卖出的金额）可由 `triggerNav` 实时算出，不冗余存。

#### 值对象 `Measure`（新增 · `@Embeddable`）

通用的"带单位量值"抽象——一个数值配它的量纲单位。本期只在 `SignalLogEntity.suggestedMeasure` 使用；如果以后别处有"值 + 单位"二选一的场景可复用同类型。

```
@Embeddable
class Measure {
  BigDecimal value;
  MeasureUnit measureUnit;
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `value` | BigDecimal | 量的数值 |
| `measureUnit` | `MeasureUnit` | 量纲：`AMOUNT`（金额，元）/ `SHARES`（份额，份） |

> **操作表已合并进 `FundTransactionEntity`**：原 `StrategyOperationEntity` 的策略上下文字段（`signalLog`）落到 `FundTransactionEntity` 上作为 nullable 字段。用户回应信号 = 写一条带策略上下文的 FundTransaction；`OVERRIDE_SKIP` 不写交易（跳过即无记录，与被动忽略在数据上不区分）；手动操作 / 转账该字段为 null。**策略动作本身不另存字段**——`operationType` 概念已删，要查"这条交易是建仓/加档几/移动止盈/逻辑止损/再平衡"通过 `signalLog.signalType` + `triggerTier` 反推即可，详见 2.3。

### 2.7 `FundStrategyActivationEntity`（基金策略激活记录 · 新增）

记录**基金启用了哪版参数、用了多久**——基金维度的激活谱系，不是参数内部状态机。一行 = 基金的一次激活任期（某参数版本从启用到停用）。

| 字段                 | 类型                       | 说明                   |
|--------------------|--------------------------|----------------------|
| 继承自 AbstractEntity |                          |                      |
| fund               | ManyToOne LAZY           | 基金                   |
| fundStrategy       | ManyToOne LAZY           | 被启用的那版参数             |
| activatedAt        | Instant                  | 启用时间                 |
| deactivatedAt      | Instant, nullable        | 停用时间；null = 当前生效     |

- **激活**：插一行（`deactivatedAt = null`）。
- **停用**：回填 `deactivatedAt`。被新版本顶替的旧 `EFFECTIVE` 参数同时回退到 `CALIBRATED`——`FundStrategyActivationEntity.deactivatedAt` 负责任期记账，`FundStrategyEntity.status` 的 `EFFECTIVE → CALIBRATED` 回退负责主表状态。
- **查当前生效**：`deactivatedAt IS NULL`（同一基金最多 1 条，由主表 `status = EFFECTIVE` 部分唯一索引保证一致）。
- **查激活历史**：按 `fund` + `activatedAt` 排序。

> `StrategyConfigService` 的 `activate` / 主动停用方法里统一处理激活记录的写入与回填，业务层不需要手动调。`activate` 不只是写新行——它还要回填上一任激活记录的 `deactivatedAt`，并将旧 `EFFECTIVE` 参数的主表状态回退到 `CALIBRATED`。
>
> **不存 `supersededBy`**：任期顶替关系由 `activatedAt` 时间序自然表达（"哪个新版本顶替了哪个旧版本"按时间排序即可），不冗余存指针。
>
> **只记启用/停用，不记参数内部状态变化**：`PENDING_CALIBRATION→CALIBRATED`、`EFFECTIVE→CALIBRATED` 等参数内部状态在主表 `status` 里够用，没有审计价值，不进激活表。
>
> `FundStatus` 变更**不**走激活表：变更不频繁，无审计需求，当前状态字段足够。如果以后真需要审计，再补对称的
`FundStatusChangeEntity` 不迟。

## 2A. 激活流程（重点）

```
PENDING_CALIBRATION
  │
  │  人工确认参数定好
  ▼
CALIBRATED  ── (calibrate 时已自动跑过去一年回测，自带一份 StrategyBacktestEntity)
  │
  │  POST /strategies/{id}/backtest  (可选：换窗口/数据更新后复跑，传回测窗口/净值序列)
  │  → 系统按该版本参数在历史净值上模拟执行
  │  → 落 StrategyBacktestEntity 一行（calibrate 时已自动跑过一次近一年窗口，这里只是复跑）
  │
  │  POST /strategies/{id}/activate
  │  → 服务先查：是否存在 passed = true 的回测结果？
  │     否 → 抛 BusinessException("NO_VALID_BACKTEST")，不激活
  │     是 → CALIBRATED → EFFECTIVE
  │        同基金旧 EFFECTIVE → 回退 CALIBRATED（不再用 RETIRED 状态）
  │
  ▼
EFFECTIVE
  │
  │  新版参数被激活 或 人工停用
  ▼
（回退到 CALIBRATED；不再用 RETIRED 状态）
```

**"通过"判定（`passed`）**：在相同回测窗口下，策略须**同时**满足：

1. 最终收益率大于三条基准线收益率（沪深300 同期、一次性 all-in、等额定投）
2. 策略最大回撤 ≤ all-in 同期最大回撤

两条都成立才算通过，`passed = true`。任一不成立 → `passed = false`，不允许激活。

> 收益率条件是策略的收益价值证明；回撤条件是纪律策略的设计底线——分档加仓的回撤不该比一次性 all-in 还大，否则逻辑破产。回撤用
`≤` 而非 `<`，避免临界值抖动导致反复激活失败。不要求回撤跑赢三条基准（定投/沪深300 回撤可能很小，门槛过高会让回测前置形同虚设）。

## 3. 枚举

### 3.1 `FundStatus`（改造现有）

```
PENDING_HOLDING   未建仓
HOLDING      持仓中
CLEARED       已清仓
```

合法跃迁：

- `PENDING_HOLDING → HOLDING`（建仓首笔确认）
- `HOLDING → CLEARED`（持仓份额归零）
- `CLEARED → PENDING_HOLDING`（人工确认继续观察，清空持有期相关的运行时状态）

> `ARCHIVED` 不作为 `FundStatus` 的值。归档通过软删除表达（见 0 节"硬性原则"和《落地讨论》1.1A）。
> `canTransitionTo` 内置在枚举里。

### 3.2 `StrategyParamStatus`（新增 · 3 态，无 `RETIRED`）

```
PENDING_CALIBRATION   待校准
CALIBRATED            已校准
EFFECTIVE            已生效
```

**没有 `RETIRED`**：被新版本顶替的旧 `EFFECTIVE` 参数直接回退到 `CALIBRATED`，不再引入退役落点状态——激活任期由激活表 `FundStrategyActivationEntity` 表达，主表状态保持精简。

合法跃迁：

- `PENDING_CALIBRATION → CALIBRATED`（人工确认校准完成 + **自动跑过去一年回测并落 `StrategyBacktestEntity`**，CALIBRATED 状态自带一份回测结果）
- `CALIBRATED → EFFECTIVE`（前置：存在一份 `passed = true` 的回测结果，跑赢三条基准线且回撤 ≤ all-in）
- `EFFECTIVE → PENDING_CALIBRATION`（被新版本顶替回退到 `CALIBRATED`；`CLEARED → PENDING_HOLDING` 时该基金所有参数统一强制回退到 `PENDING_CALIBRATION`——清仓是分水岭，旧校准和旧回测结果都不可信，必须重新走 calibrate → 自动回测）

### 3.3 `FundCategory`（新增）

```
BROAD_BASE   宽基
SECTOR       行业
ACTIVE_FUND  主动
MIXED        混合
```

> 注意 `ACTIVE_FUND` 不叫 `EFFECTIVE`，避开与 `StrategyParamStatus.EFFECTIVE` 同名歧义。
>
> `FundCategory` 落在 `FundEntity.fundCategory`，**不在 `FundStrategyEntity` 上**——它是基金本身的属性（用来查默认档位和全局硬约束上限），跟参数版本无关，换参数版本不需要重选类型。

### 3.4 `SignalType`（新增）

```
NONE    当天无信号（已评估，未触发任何动作建议）
BUILD   建仓信号
ADD     加仓信号（具体档位看 SignalLog.triggerTier）
SELL    卖出信号（具体类型由 SignalLog.reason 表达：TRAILING_STOP 移动止盈 / LOGIC_BROKEN 逻辑破坏止损 / REBALANCE 再平衡）
```

> 不再有独立的 `StrategyOperationType` 枚举。一条交易对应的策略动作（建仓 / 加档几 / 移动止盈 / 逻辑止损 / 再平衡）**不另存字段
**，要查时通过 `FundTransaction.signalLog` 关联读 `SignalLog.signalType` + `reason` + `triggerTier`
> 反推即可。用户改了建议金额下单（override）**不留痕**——交易金额就是实际值，不与建议值做差值记录；用户主动跳过信号（
`OVERRIDE_SKIP`）不写交易记录，与被动忽略在数据上不区分（要区分的话以后在 `SignalLogEntity` 加 `responseType` 字段，本期不做）。
>
> `NONE` 不是"未评估"，而是"评估过了但今天不该动"——和"未生成"区分。每日定时任务对每只 `EFFECTIVE` 基金都生成一行，含 NONE 行，作为审计轨迹。
>
> **SELL 信号优先级（2026-06-24 修订）**：`evaluateSignal` 按逻辑止损 > 移动止盈 > 再平衡顺序检查，命中即返回，一只基金每日一行
> SignalLog，最多一类 SELL 信号。
>
> **`reason` 值域**：`NO_EFFECTIVE_STRATEGY` / `TIER_HIT` / `COOLDOWN` / `WATCH_AND_WAIT` / `TRAILING_STOP` /
`LOGIC_BROKEN` / `REBALANCE` / `MIN_HOLD_DAYS_NOT_MET` / `NO_TIER_TO_SELL` / `INSUFFICIENT_MARKET_DATA` / ...

### 3.6 `NavSource`（新增）

```
MANUAL      人工录入
API_SYNC    接口同步（预留，本期不实现）
```

### 3.7 `FundTransactionSource`（原 `FundFlowSource` 改名）

沿用现有五值，不做收敛。`INCREASE` / `DECREASE` 承担买卖，语义在系统中已稳定，不强行改名为 `INCREASE` / `DECREASE` 以免做无意义改动。

```
INCREASE       买入（份额增加；金额已知、份额等净值回填）
DECREASE       卖出（份额减少；份额已知、金额等净值回填）
TRANSFER_IN    转入（份额增加；基金转换的入账腿或外部转入）
TRANSFER_OUT   转出（份额减少；基金转换的出账腿或外部转出）
INVEST         定投扣款（份额增加；本期内不实现定投计划，但枚举值保留，将来 `DcaPlanEntity` 落实时每期落一条 INVEST）
```

> 转换用 `TRANSFER_IN` + `TRANSFER_OUT` 两条交易通过 `relatedTransaction` 互指表达。`INVEST` 本期保留枚举值不使用——定投计划是另一张 `DcaPlanEntity` 表的事，启用时每期写一条 `INVEST` 交易即可。

### 3.8 `FundTransactionStatus`（改造 · 原 `FundFlowStatus` 改名 + 重做值）

```
PENDING    已下单，等净值确认（买入 amount 已知 / 卖出 shares 已知，另一侧 null）
CONFIRMED      净值确认完成（shares + amount + nav 都已回填）
CANCELLED      撤单（用户撤单；转换撤单时两条腿一起 CANCELLED）
```

> 原 `FundFlowStatus` 的值整体替换。`PENDING` 是基金交易的正常中间态——买入下单时知道金额、不知道份额（净值未公布），卖出下单时知道份额、不知道金额。`NavConfirmJob` 每晚净值公布后跑，把 `PENDING` 行回填另一侧并转 `CONFIRMED`。聚合只看 `CONFIRMED` 行。

### 3.9 `MeasureUnit`（新增 · 值对象 `Measure` 的量纲枚举）

```
AMOUNT   金额（元）
SHARES   份额（份）
```

> 配合 `Measure` 值对象使用（见 2.4 末尾值对象定义）。`SignalLog.suggestedMeasure.measureUnit` 决定 `value` 是金额还是份额。

## 4. Service 层职责切分

### 4.1 `StrategyConfigService`

- `createDraft(fundId, configRequest)` → 写一条 `PENDING_CALIBRATION`
- `updateDraft(strategyId, configRequest)`
- `calibrate(strategyId)` → `PENDING_CALIBRATION → CALIBRATED`：人工确认校准完成的同时**自动触发回测**（净值序列由
  `NavHistoryProvider` 拉取，本期接入东方财富 `pingzhongdata.js`，ADR-0002），落一行 `StrategyBacktestEntity`。回测窗口*
  *固定为"过去一年"**——不接受窗口覆盖，那是 `backtest` 接口的事。**窗口起点降级规则**：若该基金成立不满一年（`NavHistory`
  最早日期距今不足 1 年），自动降级为"自基金成立至今"，`backtestStart` 字段实际写入的是最早可用日期（不引入新字段）。CALIBRATED
  状态自带一份回测结果，`passed = true` 可直接 `activate`，`passed = false` 则提示用户调参数（回 `PENDING_CALIBRATION`
  ）或换窗口重测。**`passed` 判定跟窗口长度脱钩**——平台不替用户判断窗口够不够长，`backtestStart`
  字段本身就是窗口长度的客观表达，用户自己决定要不要信任结果。
- `backtest(strategyId, BacktestRequest)` → 可选：换窗口或数据更新后复跑回测，结果包含三条基准线对比（calibrate 时已自动跑过一次近一年窗口，本方法用于人工覆盖窗口或复跑）
- `activate(strategyId)` → 校验该版本已通过回测且跑赢三条基准线，通过后 `CALIBRATED → EFFECTIVE`，同基金旧 `EFFECTIVE` 转
  回退 `CALIBRATED`（同时回填激活表上一任 `deactivatedAt`）
- `retire(strategyId)` → 主动停用：`EFFECTIVE → CALIBRATED`，回填 `FundStrategyActivationEntity.deactivatedAt`
- 查询：`findActive(fundId)`、`listByFund(fundId)`、`getBacktestResult(strategyId)`

> 激活的硬性前置条件：参数必须先通过回测验证，跑赢三条基准线（沪深300、一次性 all-in、等额定投）才允许 `activate`
> 。回测结果落表，便于追溯和复跑。
>
> **激活/停用写激活表**：`activate` 写一行 `FundStrategyActivationEntity`（`deactivatedAt = null`），同时回填上一任激活记录的 `deactivatedAt`，并把旧 `EFFECTIVE` 参数主表状态回退到 `CALIBRATED`。主动停用方法只回填 `deactivatedAt`。`calibrate` 不写激活表（参数内部状态变化不记）。业务层不需要手动调。
>
> **`CLEARED → PENDING_HOLDING` 时的参数处理**：清仓后人工确认重新观察，基金回 `PENDING_HOLDING` 时，**该基金下所有参数版本统一回退到 `PENDING_CALIBRATION`**（清仓是分水岭，旧校准和旧回测结果都不可信，必须重新走 calibrate → 自动回测才能再到 `CALIBRATED`/`EFFECTIVE`）。回填激活表所有未停用任期的 `deactivatedAt`。

### 4.2 `FundPositionService`

按以下规则给出某只基金当前的"事实仓位"，第一版实时聚合不缓存：

- **持仓份额** = `Σ FundTransactionEntity.shares × direction`（`status = CONFIRMED` 的交易；`PENDING` 算在途，不进持仓）
- **在途订单** = `Σ ... WHERE status = 'PENDING'`（前端单独展示"已下单待净值确认"）
- **持仓金额** = 持仓份额 × 最新净值（净值由调用方传入，本期不接行情源）
- **持仓占比** = 持仓金额 / 总权益仓位
- **持仓成本** = `Σ amount × direction`（CONFIRMED 状态；不算盈亏，只是名义投入）

direction: `INCREASE`/`TRANSFER_IN` = +1，`DECREASE`/`TRANSFER_OUT` = -1。不存任何中间结果。性能成问题时再加快照表。

### 4.3 `DisciplineStrategyService`

信号引擎。两类入口分开：**生成**（定时任务调）和**确认**（用户调）。信号引擎核心算法是无状态纯函数，两类入口都复用它。

```
generateDailySignals(signalDate)                     # 定时任务入口：14:50 跑，遍历所有 EFFECTIVE 基金
  → 对每只基金调 evaluateSignal(fund, strategy, marketIndicators, capitalContext)
  → 落一行 SignalLogEntity（含 NONE）
confirmOperation(signalLogId, confirmRequest)        # 用户回应入口：写 FundTransaction + 推进状态（OVERRIDE_SKIP 不写交易）
```

**`evaluateSignal` 的固定步骤**（纯函数，输入 `FundEntity` + `FundStrategyEntity` + 市场指标 + 资金上下文，输出信号结果）：

```
1. 校验 FundStatus 是否允许出该类信号（见组合表）；不允许 → 返回 NONE
2. 校验 EFFECTIVE 策略存在；没有 → 返回 NONE，reason=NO_EFFECTIVE_STRATEGY
3. 实时派生回撤基准（ADR-0001）：
   - peakNav = max(fund_nav_history.accumulated_nav)            ← 加仓档位用
   - holdingPeriodPeakNav = 同上 + WHERE nav_date >= openedAt   ← 移动止盈用
4. 算当前回撤：(peakNav - 最近确认净值) / peakNav
5. 反弹清空判断（加仓信号专属，含 0.5% 缓冲带 ADR-0003）：
   for N in [1, 2, 3, 4]:
     if tierNAddedAt is not null AND drawdown < tierNDrawdown - 0.005:
       清空 tierN 及更深档的 AddedAt
       warnings.add("TIER_CLEARED: ...")
6. 算策略动作：
   - BUILD：FundStatus = PENDING_HOLDING AND 三条件全满足（年线上方 + 年线向上 + 今天就是 60 日新高）
     → suggestedMeasure = {plannedTotalAmount × 0.10 (BUILD_RATIO), AMOUNT}
   - ADD：FundStatus = HOLDING AND 当前档位有 EFFECTIVE + 该档未加
     → suggestedMeasure = {plannedTotalAmount × tierNRatio × 最终系数, AMOUNT}（系数见 §4.5）
   - SELL：按优先级顺序查（命中即返回）：
     a) 逻辑止损（按 fundSubType 分派）
     b) 移动止盈：档位 = floor(从 holdingPeriodPeakNav 回落 / stopLossPullbackPercent)；空档跳过（B1）；卖该档加仓交易的实际入账份额（A1）
     c) 再平衡：单只占比 > singlePositionLimit
7. 跑强提示判定（BREAKDOWN_WATCH、WEEKLY_COOLDOWN、INSUFFICIENT_DATA_FOR_COOLDOWN 等）
8. 跑硬约束截断（含 5 交易日 MIN_HOLD_DAYS；逻辑止损豁免）
9. 组装结果：signalType、reason、triggerTier、coefficient、suggestedMeasure、warnings[]、hardConstraintBreaches[]
```

**`generateDailySignals` 的职责**：

- 遍历所有"绑定 EFFECTIVE 策略 + 未软删"的基金
- 通过 `MarketIndicatorProvider` 拉每只基金当天的市场指标（从 `market_indicator_snapshot` 表读，数据由
  `MarketDataFetchJob` 14:30/14:40/14:50 分批从东方财富拉取）
- 通过 `FundPositionService` 算每只基金的资金上下文（用于硬约束）
- 调 `evaluateSignal`，结果落 `SignalLogEntity`（一行/基金/日，含 NONE）
- 唯一性约束兜底重复执行：同一 `(fund_id, signal_date)` 已存在则跳过或覆盖（按业务定，倾向覆盖以支持重跑）

**`confirmOperation` 的推进逻辑**（按 SignalLog 的 signalType + triggerTier 分派；不允许传 operationType）

| SignalLog 类型                                                          | 推进动作                                                                                                                                                                                                                                       |
|-----------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `signalType=BUILD`                                                    | 写 FundTransaction（`source=INCREASE`，`amount` 已知，`shares=null`，`status=PENDING`；`amount` 用用户实际下单值，与建议金额不一致也不留痕）；FundEntity.status = HOLDING；FundEntity.openedAt = now                                                                       |
| `signalType=ADD, triggerTier=N`                                       | 写 FundTransaction（`source=INCREASE`，同上；`amount` 用用户实际下单值，与建议金额不一致也不留痕）；FundStrategy.tierNAddedAt = now                                                                                                                                     |
| `signalType=SELL`（reason 区分 TRAILING_STOP / LOGIC_BROKEN / REBALANCE） | 写 FundTransaction（`source=DECREASE`，`shares` 已知，`amount=null`，`status=PENDING`；`shares` 用用户实际下单值，与建议份额不一致也不留痕）；TRAILING_STOP 时清对应档的 `tierNAddedAt`；LOGIC_BROKEN 时清全部 `tier1~4AddedAt`；如果聚合后 CONFIRMED 持仓份额归零 → FundEntity.status = CLEARED |
| 用户主动跳过 / 不回应                                                          | 不写 FundTransaction，不推进任何状态（被动忽略与主动跳过在数据上不区分）                                                                                                                                                                                               |                                                          |

> 买入写 `amount` 不写 `shares`，卖出写 `shares` 不写 `amount`——另一侧等 `NavConfirmJob` 当晚净值确认后回填，状态从 `PENDING` 转 `CONFIRMED`（见 4.5）。
>
> 建议侧字段（triggerNav / triggerTier / coefficient / suggestedMeasure）已在 `SignalLogEntity` 里冻好，`FundTransaction` 通过 `signalLog` 关联即可拿到，不在交易表里冗余。`ConfirmOperationRequest` 因此不用带 `MarketIndicator` / `CapitalContext`，确认时无需复算信号。

**关键约束：查询接口只读、生成任务才写**

- `evaluateSignal` 是纯函数，本身不读写数据库
- `SignalLogEntity` 的写入只发生在 `generateDailySignals` 里（含步骤 4 的反弹清空对 `tierNAddedAt` 的写——它是状态机自动行为，跟着生成走）
- 档位"已加"标记（`tierNAddedAt`）的另一种写入发生在 `confirmOperation` 里
- 查询接口（`GET /signals/...`、`GET /signals/today` 等）只读 `SignalLogEntity`，不触发任何计算或写入
- 加仓建议金额 = `plannedTotalAmount × tierNRatio × 最终系数`，再被硬约束截断（在 `evaluateSignal` 里算）

### 4.4 `NavConfirmJob`

每晚基金公司公布当日净值后跑（一般 20:00-21:00），把当天 `PENDING` 的 `FundTransactionEntity` 行回填齐并转 `CONFIRMED`。

```
run()
  1. 查所有 status = PENDING 且 fund 今日有 NavHistory 行的交易
  2. 对每条：
     - 查 FundNavHistoryEntity 拿当日 nav
     - source = INCREASE  → shares = amount / nav
     - source = DECREASE → amount = shares × nav
     - 填 nav、confirmTime = now，status = CONFIRMED
  3. 撤单窗口外的 PENDING 行保留（基金公司未公布净值的边缘情况），等次日 job 再处理
```

> 用户撤单走另一个路径：`POST /api/transactions/{id}/cancel` 把 `PENDING` 行转 `CANCELLED`（基金转换撤单时两条 `relatedTransaction` 一起 CANCELLED）。`CONFIRMED` 行不可撤。

### 4.5 `support/` 纯函数工具

- `DefaultTierTable`：宽基/行业/主动/混合 × 四档的**默认回撤阈值 + 默认加仓比例**表（建仓 PENDING_CALIBRATION
  时填默认值用）。回撤阈值取自框架第四节，四档加仓比例四类基金统一用金字塔递增（深档加更多），合计 90%：

  | 类型 | 一档回撤 | 二档回撤 | 三档回撤 | 四档回撤 | 一档比例 | 二档比例 | 三档比例 | 四档比例 |
    |---|---|---|---|---|---|---|---|---|
  | 宽基 | -8% | -15% | -25% | -35% | 15% | 20% | 25% | 30% |
  | 行业 | -15% | -25% | -35% | -45% | 15% | 20% | 25% | 30% |
  | 主动 | -12% | -20% | -30% | -40% | 15% | 20% | 25% | 30% |
  | 混合 | -12% | -22% | -32% | -40% | 15% | 20% | 25% | 30% |

  > 建仓首笔 10% + 四档 90% = 100% 计划总仓位，正好用满。差异化体现在回撤阈值上，比例不分类型调。

- `DefaultCoolDownTable`：单周跌幅冷静阈值默认值，按 `FundCategory`：宽基 8% / 行业 12% / 主动 10% / 混合 10%
- `CoefficientTable`：年线/MACD/成交量 → 系数的查表（框架 §七 原文照抄）
    - 年线 3 档：上方且向上 = 1.0 / 上方但向下 = 0.7 / 下方且向下 = 0.4
    - 周 MACD 4 档：底背离 = 1.2 / 绿柱缩小 = 1.0 / 红柱缩小 = 0.9 / 绿柱扩大 = 0.6
    - 成交量 3 档：地量企稳 = 1.2 / 正常 = 1.0 / 放量下跌 = 0.5
- `CoefficientCombiner`：合成 + clamp(0.3, 1.5)
- `WeeklyDropCalculator`：单周跌幅算法（A1 两点跌幅 + B1 含今日的最近 5 日 + C2 用累计净值 + D2 数据不足降级）；加仓冷静 +
  主动基金逻辑止损共用
- `HardConstraintConfig`：**全局硬约束规则**（代码常量），跟框架第九节一致：
    - `BUILD_RATIO = 0.10`（建仓首笔比例，框架固定纪律）
    - `TIER_CLEAR_BUFFER = 0.005`（反弹清空缓冲带 0.5%，ADR-0003）
    - `singlePositionLimit`（按 FundCategory）：宽基/主动/混合 20%，行业 15%
    - `categoryPositionLimit`：单类型上限 30%（同 FundCategory 汇总）
    - `totalEquityPositionLimit`：总权益仓位上限 80%（分母取 `UserConfig.totalInvestableCapital`）
    - `singleAddRatioLimit`：单次加仓 ≤ 该品种剩余可用仓位的 50%
    - `minHoldDays`：建仓后 **5 个交易日**内不赎回（逻辑破坏止损豁免；起算 = `max(openedAt, tier1~4AddedAt)`）
- `HardConstraintChecker`：五条硬约束统一入口，调用 `HardConstraintConfig` 读上限，调用 `TradingCalendarService` 判定 5 交易日

放静态工具类或 `@Component` 都可以，倾向静态——它们都是纯函数。

## 5. DTO 设计

### 5.1 请求 DTO（关键的几条）

`StrategyConfigRequest`

- 5 个参数字段（四档回撤阈值 + 四档加仓比例 + 单周冷静阈值 + 移动止盈回落幅度）；同期更新的 `FundEntity.fundCategory`、`FundEntity.plannedTotalAmount`（基金属性由本请求一并落地）

`MarketIndicators`（信号生成入参，由 `MarketIndicatorProvider` 在 14:50 从 `market_indicator_snapshot` 表读取；数据来源是东方财富
14:30/14:40/14:50 分批拉取，ADR-0002）

- `currentNav`（可空，传则覆盖记录中的最近确认净值；`peakNav` 不再作为入参，由 `fund_nav_history` 实时派生）
- `priceAboveYearLine` / `yearLineRising`（推年线状态）
- `weeklyMacdState`（enum：底背离 / 绿柱缩小 / 红柱缩小 / 绿柱扩大）
- `volumeState`（enum：地量 / 正常 / 放量下跌）
- `weeklyDropPercent`
- `isSixtyDayHigh`（仅建仓用）

`CapitalContext`

- `totalInvestableCapital`、`currentTotalEquityPosition`、`currentCategoryPosition`（用于跑硬约束）

`ConfirmOperationRequest`

- `signalLogId`（必填，指向被回应的那条 `SignalLogEntity`）
- `actualAmount`（买入类操作必填：对应 `signalType=BUILD` 或 `signalType=ADD`；买入下单时知道金额、不知道份额；用户可填与建议金额不同的实际值，系统不留痕）
- `actualShares`（卖出类操作必填：对应 `signalType=SELL`；卖出下单时知道份额、不知道金额；用户可填与建议份额不同的实际值，系统不留痕）

> 买入传金额、卖出传份额——对应基金交易的现实时序，用户下单时确实知道哪一侧。另一侧等 `NavConfirmJob` 当晚净值确认后回填。
>
> 不再需要在请求里带 `MarketIndicators` / `CapitalContext` / `actualNav` / `overrideReason`——建议侧字段已在 `SignalLogEntity` 里冻好，确认时直接通过 `signalLogId` 关联读取，不复算信号；`actualNav` 在下单时根本拿不到（净值未公布）；`overrideReason` 已删除（用户改了建议金额下单不留痕，交易金额就是实际值，不需要专门字段记录是否 override、也不做差值留痕）。
>
> 用户主动跳过信号（`OVERRIDE_SKIP`）不调本接口、不写交易——与"被动忽略"在数据上不区分。

### 5.2 响应 DTO

信号视图直接对应 `SignalLogEntity` 行（查询接口返回），不再有"算一次返回一次"的临时响应：

```
SignalView {
  signalLogId: Long
  fundId: Long
  signalDate: Instant
  signalType: SignalType        // NONE / BUILD / ADD / SELL
  reason: String?               // NO_EFFECTIVE_STRATEGY / TIER_HIT / COOLDOWN / WATCH_AND_WAIT / TRAILING_STOP / LOGIC_BROKEN / REBALANCE / MIN_HOLD_DAYS_NOT_MET / NO_TIER_TO_SELL / INSUFFICIENT_MARKET_DATA / ...
  triggerNav: BigDecimal?
  triggerTier: Integer?         // ADD 或 SELL(TRAILING_STOP) 时
  coefficient: BigDecimal?
  suggestedMeasure: Measure?   // 系统建议量 + 单位；BUILD/ADD 时 {value, AMOUNT}（建议金额），SELL 时 {value, SHARES}（建议份额）；NONE 时为 null
  warnings: List<Warning>       // 强提示：强制冷静、破位观望、反弹清空 TIER_CLEARED 等
  hardConstraintBreaches: List<Breach>  // 命中硬约束被截断的说明
  respondedAt: Instant?         // 已回应时指向 FundTransaction.confirmTime（或 createdDate，下单时刻）；null = 未回应（被动忽略或主动跳过，两者不区分）
}

    - 买入 `PENDING` 行 → `shares = amount / nav` 回填、`status=CONFIRMED`
    - 卖出 `PENDING` 行 → `amount = shares × nav` 回填、`status=CONFIRMED`
    - 当日无净值行的 `PENDING` 保留不变（等次日再跑）
    - 撤单：`PENDING → CANCELLED`；转换撤单时两条 `relatedTransaction` 一起 CANCELLED
- **回测引擎**：单测固定净值序列场景，验证策略收益计算、三条基准线对比（收益 + 回撤）、`passed` 判定（含边界：收益刚好相等、回撤刚好等于 all-in、收益跑赢三条但回撤超 all-in 等）。
- **FundPosition 聚合**：构造多笔 `FundTransactionEntity`（含 `PENDING` / `CONFIRMED` / `CANCELLED` 混合、份额增减、转换两条腿），验证持仓份额 = Σ shares × direction（仅 CONFIRMED）、在途订单 = Σ PENDING、CANCELLED 不计入、持仓金额随传入净值变化。
- **不跑**端到端 Web 测试（这一期）。
