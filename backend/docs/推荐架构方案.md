# 基金纪律策略 · 推荐架构方案

> 本文档只写"我倾向怎么做"，目的是给讨论一个具体的靶子，方便逐点拍板或反对。所有决策点在《落地讨论-策略实现.md》里有对应的开放问题。两份文档讨论完后合并为最终设计文档。

## 0. 设计取向

- **薄信号引擎、厚事实账目**：策略服务只做"看一眼当前状态 + 给一条建议"，不存计算缓存；事实数据（资金流水、操作记录）老老实实落表。
- **单一事实源：账目只存份额，金额永远实时算**（见下方硬性原则）。
- **配置与状态同表**：策略参数和该参数下的运行时状态（前高、四档时间戳等）放同一张表，按 `paramStatus` 区分版本。换参数 =
  新建一份，不是改旧的。
- **市场指标外部化**：年线/MACD/成交量等本期不接行情源，全部作为请求入参传入，留接口占位。
- **人工执行、系统记账**：系统出建议，人工拍板；拍板后由系统统一写流水与操作记录，保证两条账对得上。

### 硬性原则：账目只存份额，金额永远实时算

**写入侧（事实账目，不可变）**

- `FundFlowEntity.shares` —— **事实**，每笔买卖确认后永不改变
- `FundFlowEntity.amount` —— 成交瞬时金额快照，仅用于审计和反推成交净值，**不参与任何后续计算**
- `fund_nav_history` —— 基金净值历史表，只读事实数据

**读取侧（展示/计算，永远实时算，永不落库）**

- 持仓份额 = `Σ shares × direction`（CONFIRMED 状态流水）
- 持仓金额 = `持仓份额 × 最近一次确认净值`
- 持仓占比 = `持仓金额 / 总权益持仓金额`
- 浮动盈亏 = `持仓金额 - Σ amount × direction`
- 当前回撤 = `(前高净值 - 最近确认净值) / 前高净值`

**纪律意图（不是事实账目，必须存金额）**

- `UserFundStrategyEntity.plannedTotalAmount`（计划总仓位）
- `tier1Amount ~ tier4Amount`（四档基础加仓额）
- `singlePositionLimit`（单品种仓位上限）

> 准确表述：**事实账目（流水、净值快照）只存份额和成交瞬时金额；策略意图（计划仓位、加仓额）存金额；展示用的当前持仓金额永远不落库，永远用
`份额 × 当前净值` 实时算。**

**为什么这么做**

1. 单一事实源：账目只有"买了多少份、什么时候买的"这种客观事实；"现在值多少钱"是派生信息，跟着净值走。两边永远不会矛盾。
2. 净值变动零成本：今天净值涨 3%，不需要写任何数据、不跑任何 job，查询时自然反映。如果维护金额，得每天跑 job 刷所有持仓的
   `amount` 字段，跑挂了账目就错。
3. 历史可复现：任何一天的市值都能算（那天的份额 × 那天的净值）。如果存金额，历史就锁死成"那天跑 job 时算出的值"，跑漏一天丢一天。
4. 净值修正能自然吸收：基金公司偶尔修正前日净值（分红除权、错账修正等）。存金额的话所有历史展示都得重算；份额制下只要修正净值表那一行，所有展示自动正确。

### 硬性原则：归档 = 软删除，与持仓状态正交

归档的真实场景是"我不想在这个平台看见这只基金了"（上错了、不想再看了、没买过也不打算买）。这不是持仓状态的一部分，直接用
`AbstractEntity` 现有的 `@SoftDelete` + `deletedDate` 表达：

- **归档动作** = 执行软删除（`deletedDate = now`）
- **取消归档** = 恢复（`deletedDate = null`），状态回到归档前的样子
- 默认查询天然过滤已软删记录（Hibernate `@SoftDelete` 自动加 `WHERE deleted_date IS NULL`）
- 不引入 `visibility` / `archived` 这种冗余字段

**关键边界：**

1. **任意 `FundStatus` 都允许软删**，包括 `HOLDING`。不卡校验、不强制填原因——用户可以强制归档仍有持仓的基金。
2. **归档时关联数据一起软删**：`FundFlowEntity`、`StrategyOperationEntity`、`UserFundStrategyEntity`、`FundNavHistoryEntity`
   中该基金的所有关联记录同步软删。恢复归档时一并恢复。理由：归档是"我不想看见这只基金了"
   ，关联数据是这只基金的衍生信息，跟着基金一起隐藏最自然，不留半残数据。
3. **软删 ≠ 物理删除**：数据物理保留，只是从视图隐藏。物理清理过期归档数据另起独立任务，不混进归档动作。

## 2. 实体设计

### 2.1 `FundEntity`（基金产品）

只描述"是哪只基金" + 跟策略无关的**基金属性和市场事实 / 持有事实**，不存仓位事实。

| 字段                   | 类型                        | 说明                                                                |
|----------------------|---------------------------|-------------------------------------------------------------------|
| 继承自 AbstractEntity   |                           | id / version / 审计 / 软删                                            |
| fundCode             | String                    |                                                                   |
| fundName             | String                    |                                                                   |
| investmentTarget     | InvestmentTarget enum     | 已有                                                                |
| operationMode        | OperationMode enum        | 已有                                                                |
| investmentPhilosophy | InvestmentPhilosophy enum | 已有                                                                |
| status               | **FundStatus**（重做）        | 未建仓 / 持仓中 / 已清仓                                                   |
| plannedTotalAmount   | BigDecimal, nullable      | 计划总仓位（纪律意图）；建仓时（NOT_OPENED → HOLDING）必须填，HOLDING/CLOSED 状态下非 null |
| peakNav              | BigDecimal                | 前高净值（基金历史最高累计净值，市场事实，每月更新）                                        |
| holdingPeriodPeakNav | BigDecimal, nullable      | 持有期最高净值（我买入后的最高净值，算移动止盈用）；未建仓时为 null                              |
| openedAt             | Instant, nullable         | 首次建仓时间（持有事实，跟策略版本无关）；`CLOSED → NOT_OPENED` 时清空                    |

**移除**：现有的 `amount` 和 `shares` 字段。事实仓位由 `FundFlowEntity` 聚合得出，不在产品实体上冗余。

> `plannedTotalAmount` / `peakNav` / `holdingPeriodPeakNav` / `openedAt` 都从 `UserFundStrategyEntity`
> 搬过来——它们跟策略参数版本无关，是基金本身的属性、市场事实或持有事实。换参数版本不影响这些值。

> **`FundStatus` 变更不走日志表**：变更不频繁（一只基金一辈子也就 NOT_OPENED → HOLDING → CLOSED → 偶尔回
> NOT_OPENED），没有审计/复盘需求，当前状态字段足够。如果以后真需要审计，再补日志表不迟。

### 2.2 `UserFundStrategyEntity`（策略参数 + 该版本下的运行时状态）

一份记录 = 一份策略快照。改参数 = 新建一份 DRAFT，激活时旧的转 RETIRED。

只放跟**策略参数版本相关**的字段。基金属性（`plannedTotalAmount`、`peakNav`、`holdingPeriodPeakNav`、`openedAt`）和全局硬约束（
`singlePositionLimit`）不在这里。

| 字段                            | 类型                        | 说明                                                                       |
|-------------------------------|---------------------------|--------------------------------------------------------------------------|
| 继承自 AbstractEntity            |                           |                                                                          |
| fund                          | ManyToOne FundEntity LAZY |                                                                          |
| userId                        | Long, nullable            | 预留，本期不用                                                                  |
| paramStatus                   | StrategyParamStatus enum  | 待校准 / 已校准 / 已生效 / 已退役                                                    |
| fundCategory                  | FundCategory enum         | 宽基 / 行业 / 主动 / 混合                                                        |
| tier1Drawdown ~ tier4Drawdown | BigDecimal                | 四档回撤阈值（百分比，如 0.08 = 8%）                                                  |
| tier1Ratio ~ tier4Ratio       | BigDecimal                | 四档加仓比例（占 plannedTotalAmount 的百分比，如 0.15 = 15%）；四档之和 ≤ 90%                |
| weeklyCoolDownThreshold       | BigDecimal                | 单周跌幅冷静阈值（百分比，如 0.10 = 10%），用于强制冷静                                        |
| stopLossPullbackPercent       | BigDecimal                | 移动止盈回落幅度（百分比，如 0.08 = 8%）                                                |
| tier1AddedAt ~ tier4AddedAt   | Instant, nullable         | 方案B 四档**实际执行**时间戳，null = 未加。**只在用户通过 POST /operations 确认执行时写入**，信号检查只读不写 |

> **不在主表存 `activatedAt` / `retiredAt`**：参数状态变更走日志表 `StrategyParamStatusChangeEntity`（见 2.7），主表只保留当前状态
`paramStatus`。
> **不在主表存 `openedAt`**：搬到 `FundEntity`（持有事实，跟策略版本无关）。

**tier1~4AddedAt 写入时机**（重要）：

- 信号检查（`checkXXXSignal`）只看不写
- `POST /operations` 确认执行时才写
- operationType = `ADD_TIER_1` 写 `tier1AddedAt`，依次类推
- 用户不执行 → `tierNAddedAt` 仍是 null → 下次还在该档会再提示

**唯一性约束**：同一 `fund_id` 在未软删的记录里，最多 1 份 `paramStatus = ACTIVE`。Postgres 部分唯一索引：

```sql
CREATE UNIQUE INDEX uq_user_fund_strategy_active
    ON user_fund_strategy (fund_id) WHERE param_status = 'ACTIVE' AND deleted_date IS NULL;
```

### 2.3 `FundFlowEntity`（资金流水）

保留现状，但**必须加 `shares`（份额）字段**。

关键原则：**流水维护的是份额，不是金额**。金额会随净值每日变动，但份额只在买卖时才变。因此：

- `shares`（BigDecimal，正数）—— 事实数据，聚合用它
- `amount`（BigDecimal，正数）—— 成交时金额快照，用于审计和反推成交净值（`成交净值 = amount / shares`）
- 方向仍由 `source` 区分：`INCREASE` / `TRANSFER_IN` / `INVEST` 为份额增加，`DECREASE` / `TRANSFER_OUT` 为份额减少

持仓计算：

```
持仓份额 = Σ( shares × direction )           # direction: +1 / -1
持仓金额 = 持仓份额 × 当前净值                # 实时算，不存
持仓占比 = 持仓金额 / 总权益仓位
```

> 今天净值上涨导致"钱变多了"——不需要任何流水，只是 `持仓份额 × 新净值` 自然变大。只有真正发生买卖时才写流水。

### 2.5 `StrategyBacktestEntity`（回测结果 · 新增）

`CALIBRATED → ACTIVE` 的硬性前置：必须有一份**有效**的回测结果跑赢三条基准线。

| 字段                   | 类型             | 说明                 |
|----------------------|----------------|--------------------|
| 继承自 AbstractEntity   |                |                    |
| userFundStrategy     | ManyToOne LAZY | 被回测的那一版参数          |
| backtestStart        | LocalDate      | 回测起点               |
| backtestEnd          | LocalDate      | 回测终点               |
| initialCapital       | BigDecimal     | 初始资金               |
| strategyReturn       | BigDecimal     | 策略最终收益率            |
| strategyMaxDrawdown  | BigDecimal     | 策略期间最大回撤           |
| benchmarkHs300Return | BigDecimal     | 沪深300 同期收益率        |
| benchmarkAllInReturn | BigDecimal     | 一次性 all-in 同期收益率   |
| benchmarkDcaReturn   | BigDecimal     | 等额定投同期收益率          |
| beatsAllBenchmarks   | boolean        | 是否同时跑赢三条           |
| backtestDataSnapshot | JSONB / TEXT   | 回测用的净值序列快照，便于复跑和审计 |
| executedAt           | Instant        | 执行时间               |

> 默认回测窗口"近一年"，但允许人工指定起止日期，便于跨周期对比。

> 回测结果与参数版本一对多：同一版参数可以多次回测（窗口不同/数据更新），但只要有**任意一份** `beatsAllBenchmarks = true`
> 的结果，就允许激活。

### 2.6 `FundNavHistoryEntity`（基金净值历史 · 新增）

展示和回测都依赖"最近一次确认净值"和历史净值序列，必须有一张表承载。本期由人工/调用方录入，预留未来对接行情接口。

| 字段                 | 类型                        | 说明                        |
|--------------------|---------------------------|---------------------------|
| 继承自 AbstractEntity |                           |                           |
| fund               | ManyToOne FundEntity LAZY |                           |
| navDate            | LocalDate                 | 净值日期，业务唯一键                |
| nav                | BigDecimal                | 单位净值                      |
| accumulatedNav     | BigDecimal                | 累计净值（复权），算前高用             |
| source             | NavSource enum            | `MANUAL` / `API_SYNC`（预留） |

**唯一性约束**：`(fund_id, nav_date)` 在未软删的记录里唯一。

CREATE UNIQUE INDEX uq_fund_nav_history

```

> 接东方财富/天天基金等数据源时，只是新增一条 `source = API_SYNC` 的写入路径，业务代码全不动。

### 2.7 `StrategyParamStatusChangeEntity`（策略参数状态变更日志 · 新增）

`UserFundStrategyEntity` 主表不存 `activatedAt` / `retiredAt`，状态变更走日志表。每次 `paramStatus` 变化必须写一条。

| 字段                      | 类型                       | 说明                                              |
|-------------------------|--------------------------|-------------------------------------------------|
| 继承自 AbstractEntity      |                          |                                                 |
| userFundStrategy        | ManyToOne LAZY           | 哪一版参数                                           |
| fund                    | ManyToOne LAZY           | 冗余存一份，便于按基金查日志                                  |
| fromStatus              | StrategyParamStatus      | 变更前状态                                           |
| toStatus                | StrategyParamStatus      | 变更后状态                                           |
| triggerUserFundStrategy | ManyToOne LAZY, nullable | 触发本次变更的另一版参数（如新激活把旧版顶下来 → 旧版那条记录里 trigger 指向新版） |
| reason                  | String, nullable         | 变更原因；正常推进可不填，主动决策（RETIRED）建议填                   |
| changedAt               | Instant                  | 变更时间                                            |

> `StrategyConfigService` 的 `calibrate` / `activate` / `retire` 方法里统一处理日志写入，业务层不需要手动调。
>
> `FundStatus` 变更**不**走日志表：变更不频繁，无审计需求，当前状态字段足够。如果以后真需要审计，再补对称的
`FundStatusChangeEntity` 不迟。

## 2A. 激活流程（重点）

```
DRAFT
  │
  │  人工确认参数定好
  ▼
CALIBRATED  ── (尚未回测)
  │
  │  POST /strategies/{id}/backtest  (传回测窗口/初始资金/净值序列)
  │  → 系统按该版本参数在历史净值上模拟执行
  │  → 落 StrategyBacktestEntity 一行
  │
  │  POST /strategies/{id}/activate
  │  → 服务先查：是否存在 beatsAllBenchmarks = true 的回测结果？
  │     否 → 抛 BusinessException("NO_VALID_BACKTEST")，不激活
  │     是 → CALIBRATED → ACTIVE
  │        同基金旧 ACTIVE → RETIRED
  │
  ▼
ACTIVE
  │
  │  新版参数被激活 或 人工停用
  ▼
RETIRED
```

**"跑赢三条基准线"判定**：在相同回测窗口和初始资金下，策略最终收益率须**同时**大于：

1. 沪深300 同期收益率
2. 一次性 all-in（期初全部买入并持有至期末）收益率
3. 等额定投（将初始资金按交易日数等分，每日/每周定投）收益率

三条同时成立才算有效。任一不成立 → `beatsAllBenchmarks = false`，不允许激活。

**审计性**：净值序列快照 `backtestDataSnapshot` 落库，便于事后复跑验证结果可复现。

## 3. 枚举

### 3.1 `FundStatus`（改造现有）

```
NOT_OPENED   未建仓
HOLDING      持仓中
CLOSED       已清仓
```

合法跃迁：

- `NOT_OPENED → HOLDING`（建仓首笔确认）
- `HOLDING → CLOSED`（持仓份额归零）
- `CLOSED → NOT_OPENED`（人工确认继续观察，清空持有期相关的运行时状态）

> `ARCHIVED` 不作为 `FundStatus` 的值。归档通过软删除表达（见 0 节"硬性原则"和《落地讨论》1.1A）。
> `canTransitionTo` 内置在枚举里。

### 3.2 `StrategyParamStatus`（新增）

```
DRAFT        待校准
CALIBRATED   已校准
ACTIVE       已生效
RETIRED      已退役
```

### 3.3 `FundCategory`（新增）

```
BROAD_BASE   宽基
SECTOR       行业
ACTIVE_FUND  主动
MIXED        混合
```

> 注意 `ACTIVE_FUND` 不叫 `ACTIVE`，避开与 `StrategyParamStatus.ACTIVE` 同名歧义。

### 3.4 `StrategyOperationType`（新增）

```
OPEN                建仓

## 4. Service 层职责切分

### 4.1 `StrategyConfigService`

- `createDraft(fundId, configRequest)` → 写一条 `DRAFT`
- `updateDraft(strategyId, configRequest)`
- `calibrate(strategyId)` → `DRAFT → CALIBRATED`
- `backtest(strategyId, BacktestRequest)` → 用历史净值跑回测，结果包含三条基准线对比
- `activate(strategyId)` → 校验该版本已通过回测且跑赢三条基准线，通过后 `CALIBRATED → ACTIVE`，同基金旧 `ACTIVE` 转
  `RETIRED`
- `retire(strategyId)` → 主动停用
- 查询：`findActive(fundId)`、`listByFund(fundId)`、`getBacktestResult(strategyId)`

> 激活的硬性前置条件：参数必须先通过回测验证，跑赢三条基准线（沪深300、一次性 all-in、等额定投）才允许 `activate`
> 。回测结果落表，便于追溯和复跑。
>
> **状态变更统一写日志**：`calibrate` / `activate` / `retire` 三个方法在改 `paramStatus` 后，自动写一条
`StrategyParamStatusChangeEntity`。业务层不需要手动调日志写入。
>
> **`CLOSED → NOT_OPENED` 时的参数处理**：清仓后人工确认重新观察，基金回 `NOT_OPENED` 时，原 `ACTIVE` 参数强制退到
`CALIBRATED`（写一条日志，reason 建议填"清仓后重新激活前需重新回测验证"）。重新激活前必须重新回测。

### 4.2 `FundPositionService`

按以下规则给出某只基金当前的"事实仓位"，第一版实时聚合不缓存：

- **持仓份额** = `Σ FundFlowEntity.shares × direction`（CONFIRMED 状态的流水）
- **持仓金额** = 持仓份额 × 最新净值（净值由调用方传入，本期不接行情源）
- **持仓占比** = 持仓金额 / 总权益仓位
- **持仓成本** = `Σ amount × direction`（不算盈亏，只是名义投入）

不存任何中间结果。性能成问题时再加快照表。

### 4.3 `DisciplineStrategyService`

信号引擎主入口。统一入口契约：

```
checkBuildSignal(fundId, marketIndicators, capitalContext) → BuildSignalResponse
checkAddSignal(fundId, marketIndicators, capitalContext) → AddSignalResponse
checkSellSignal(fundId, marketIndicators, capitalContext) → SellSignalResponse
confirmOperation(fundId, confirmRequest) → 写 StrategyOperation + (可选) FundFlow + 推进状态
```

每个 `check*` 方法的固定步骤：

```
1. 查 FundEntity，校验 FundStatus 是否允许出该信号（见组合表）
2. 查 ACTIVE 的 UserFundStrategy；没有 → 返回 NO_ACTIVE_STRATEGY，结束
3. 计算当前回撤（基于 FundEntity.peakNav 和最近确认净值）
4. **反弹清空判断**（加仓信号专属）：当前档位比已加最深档浅 → 清空更深档的 tierNAddedAt
5. 计算业务信号（档位 / 系数 / 强提示 / 卖出类型）
6. 跑硬约束截断
7. 组装响应：shouldSignal、reason、suggestedAmount、warnings[]
```

**关键约束：信号检查只读不写**

- 步骤 4 的"反弹清空"是唯一例外——它是状态机的自动行为，不依赖用户操作
- 档位"已加"标记（`tierNAddedAt`）只在 `confirmOperation` 里写
- 加仓建议金额 = `plannedTotalAmount × tierNRatio × 最终系数`，再被硬约束截断

**`confirmOperation` 的推进逻辑**（按 operationType switch）：

| operationType                                     | 推进动作                                                                                                |
|---------------------------------------------------|-----------------------------------------------------------------------------------------------------|
| `OPEN`                                            | 写 FundFlow（份额增）；FundEntity.status = HOLDING；FundEntity.openedAt(策略侧) = now；初始化 holdingPeriodPeakNav |
| `ADD_TIER_1 ~ 4`                                  | 写 FundFlow（份额增）；UserFundStrategy.tierNAddedAt = now                                                 |
| `TRAILING_STOP` / `LOGIC_STOP_LOSS` / `REBALANCE` | 写 FundFlow（份额减）；如果聚合后持仓份额归零 → FundEntity.status = CLOSED                                            |
| `OVERRIDE_ADD`                                    | 同对应 `ADD_TIER_N`，但记录 overrideReason                                                                 |
| `OVERRIDE_SKIP`                                   | 只写 StrategyOperationEntity，不写 FundFlow，不推进任何状态                                                      |

### 4.4 `support/` 纯函数工具

- `DefaultTierTable`：宽基/行业/主动/混合 × 四档的**默认回撤阈值 + 默认加仓比例**表（建仓 DRAFT
  时填默认值用）。回撤阈值取自框架第四节，四档加仓比例四类基金统一用金字塔递增（深档加更多），合计 90%：

  | 类型 | 一档回撤 | 二档回撤 | 三档回撤 | 四档回撤 | 一档比例 | 二档比例 | 三档比例 | 四档比例 |
    |---|---|---|---|---|---|---|---|---|
  | 宽基 | -8% | -15% | -25% | -35% | 15% | 20% | 25% | 30% |
  | 行业 | -15% | -25% | -35% | -45% | 15% | 20% | 25% | 30% |
  | 主动 | -12% | -20% | -30% | -40% | 15% | 20% | 25% | 30% |
  | 混合 | -12% | -22% | -32% | -40% | 15% | 20% | 25% | 30% |

  > 建仓首笔 10% + 四档 90% = 100% 计划总仓位，正好用满。差异化体现在回撤阈值上，比例不分类型调。

- `DefaultCoolDownTable`：单周跌幅冷静阈值默认值，按 `FundCategory`：宽基 8% / 行业 12% / 主动 10% / 混合 10%
- `CoefficientTable`：年线/MACD/成交量 → 系数的查表
- `CoefficientCombiner`：合成 + clamp(0.3, 1.5)
- `HardConstraintConfig`：**全局硬约束规则**（代码常量），跟框架第九节一致：
    - `singlePositionLimit`（按 FundCategory）：宽基/主动/混合 20%，行业 15%
    - `categoryPositionLimit`：单类型上限 30%（同 FundCategory 汇总）
    - `totalEquityPositionLimit`：总权益仓位上限 80%
    - `singleAddRatioLimit`：单次加仓 ≤ 该品种剩余可用仓位的 50%
    - `minHoldDays`：建仓后 7 天内不赎回（逻辑破坏止损除外）
- `HardConstraintChecker`：五条硬约束统一入口，调用 `HardConstraintConfig` 读上限

放静态工具类或 `@Component` 都可以，倾向静态——它们都是纯函数。

## 5. DTO 设计

### 5.1 请求 DTO（关键的几条）

`StrategyConfigRequest`

- 5 个参数字段；`fundCategory`；`plannedTotalAmount`

`MarketIndicatorRequest`（信号检查共用）

- `currentNav`、`peakNav`（可空，传则覆盖记录中的值）
- `priceAboveYearLine` / `yearLineRising`（推年线状态）
- `weeklyMacdState`（enum：底背离 / 绿柱缩小 / 红柱缩小 / 绿柱扩大）
- `volumeState`（enum：地量 / 正常 / 放量下跌）
- `weeklyDropPercent`
- `isSixtyDayHigh`（仅建仓用）

`CapitalContext`

- `totalInvestableCapital`、`currentTotalEquityPosition`、`currentCategoryPosition`（用于跑硬约束）

`ConfirmOperationRequest`

- `operationType`、`actualAmount`、`actualNav`、`overrideReason`

### 5.2 响应 DTO

统一形状，避免前端 if 分支地狱：

```
SignalResponse {
  shouldSignal: boolean
  reason: enum                // NO_ACTIVE_STRATEGY / TIER_HIT / COOLDOWN / WATCH_AND_WAIT / ...
  triggerTier: Integer?       // 加仓时
  coefficient: BigDecimal?
  suggestedAmount: BigDecimal?
  warnings: List<Warning>     // 强提示：强制冷静、破位观望等
  hardConstraintBreaches: List<Breach>  // 命中硬约束被截断的说明
}
```
