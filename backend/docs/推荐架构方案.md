# 基金纪律策略 · 推荐架构方案

> 本文档只写"我倾向怎么做"，目的是给讨论一个具体的靶子，方便逐点拍板或反对。所有决策点在《落地讨论-策略实现.md》里有对应的开放问题。两份文档讨论完后合并为最终设计文档。

---

## 0. 设计取向

- **薄信号引擎、厚事实账目**：信号引擎本身是无状态纯函数——给当前状态 + 市场指标，出一条建议；每日 14:50 定时任务跑一遍引擎，把当天的信号快照冻进 `SignalLogEntity`（14:50 的市场指标收盘后无法复算，必须当时落库）。事实数据（资金流水、信号快照、操作记录）老老实实落表。
- **单一事实源：账目只存份额，金额永远实时算**（见下方硬性原则）。
- **配置与状态同表**：策略参数和该参数下的运行时状态（前高、四档时间戳等）放同一张表，按 `paramStatus` 区分版本。换参数 =
  新建一份，不是改旧的。
- **每日定时生成信号**：每天 14:50 对所有绑定 `ACTIVE` 策略的基金跑信号引擎，结果落 `SignalLogEntity`。用户不触发信号计算，只回应已生成的信号（执行 / override / 主动跳过）；被动忽略（信号已生成但无回应）可追踪。
- **市场指标外部化**：年线/MACD/成交量等本期不接行情源，由定时任务在 14:50 通过 `MarketIndicatorProvider` 拉取（本期可半自动灌入），留接口占位。
- **人工执行、系统记账**：系统出建议，人工拍板；拍板后由系统统一写流水与操作记录，保证两条账对得上。

### 硬性原则：账目只存份额，金额永远实时算

**写入侧（事实账目，下单时写入已知侧，净值确认后回填另一侧，回填后不再改变）**

- `FundTransactionEntity.shares` —— **事实**。买入下单时为 null（已知金额、未知净值，算不出份额），净值确认后回填；卖出下单时已知；回填后永不改变
- `FundTransactionEntity.amount` —— **事实**。买入下单时已知；卖出下单时为 null，净值确认后回填；回填后永不改变
- `FundTransactionEntity.nav` —— 成交净值，净值确认后回填；`amount / shares` 反推也一致
- `FundTransactionEntity.status` —— `PENDING_NAV`（已下单等净值确认）/ `CONFIRMED`（净值确认完成）/ `CANCELLED`（撤单）
- `fund_nav_history` —— 基金净值历史表，只读事实数据

> 基金交易的现实时序：14:50 信号 → 15:00 前下单 → 当晚基金公司公布当日净值 → 系统回填另一侧。买入知道金额不知道份额，卖出知道份额不知道金额，是事实尚未发生而非数据缺失。`NavConfirmJob` 每晚净值公布后跑，把 `PENDING_NAV` 行回填 `shares`/`amount`/`nav`，转 `CONFIRMED`。

**读取侧（展示/计算，永远实时算，永不落库）**

- 持仓份额 = `Σ shares × direction`（`status = CONFIRMED` 的交易；`PENDING_NAV` 算在途，不进持仓）
- 持仓金额 = `持仓份额 × 最近一次确认净值`
- 持仓占比 = `持仓金额 / 总权益持仓金额`
- 浮动盈亏 = `持仓金额 - Σ amount × direction`
- 当前回撤 = `(前高净值 - 最近确认净值) / 前高净值`

**纪律意图（不是事实账目，必须存金额）**

- `UserFundStrategyEntity.plannedTotalAmount`（计划总仓位）
- `tier1Amount ~ tier4Amount`（四档基础加仓额）
- `singlePositionLimit`（单品种仓位上限）

> 准确表述：**事实账目（交易、净值快照）下单时写入已知侧（买入金额 / 卖出份额），净值确认后回填另一侧，回填后永不改变；策略意图（计划仓位、加仓额）存金额；展示用的当前持仓金额永远不落库，永远用
`份额 × 当前净值` 实时算。**

**为什么这么做**

1. 单一事实源：账目只有"买了多少份、什么时候买的"这种客观事实；"现在值多少钱"是派生信息，跟着净值走。两边永远不会矛盾。
2. 净值变动零成本：今天净值涨 3%，不需要写任何数据、不跑任何 job，查询时自然反映。如果维护金额，得每天跑 job 刷所有持仓的
   `amount` 字段，跑挂了账目就错。
3. 历史可复现：任何一天的市值都能算（那天的份额 × 那天的净值）。如果存金额，历史就锁死成"那天跑 job 时算出的值"，跑漏一天丢一天。
4. 净值修正能自然吸收：基金公司偶尔修正前日净值（分红除权、错账修正等）。存金额的话所有历史展示都得重算；份额制下只要修正净值表那一行，所有展示自动正确。

### 硬性原则：归档 = 软删除，与持仓状态正交

归档的真实场景是"我不想在这个平台看见这只基金了"（上错了、不想再看了、没买过也不打算买）。这不是持仓状态的一部分，直接用
`AbstractEntity` 现有的 `@SoftDelete` + `deletedDate` 表达：

- **归档动作** = 执行软删除（`deletedDate = now`）
- **取消归档** = 恢复（`deletedDate = null`），状态回到归档前的样子
- 默认查询天然过滤已软删记录（Hibernate `@SoftDelete` 自动加 `WHERE deleted_date IS NULL`）
- 不引入 `visibility` / `archived` 这种冗余字段

**关键边界：**

1. **任意 `FundStatus` 都允许软删**，包括 `HOLDING`。不卡校验、不强制填原因——用户可以强制归档仍有持仓的基金。
2. **归档时关联数据一起软删**：`FundTransactionEntity`、`SignalLogEntity`、`UserFundStrategyEntity`、`FundNavHistoryEntity`
   中该基金的所有关联记录同步软删。恢复归档时一并恢复。理由：归档是"我不想看见这只基金了"
   ，关联数据是这只基金的衍生信息，跟着基金一起隐藏最自然，不留半残数据。
3. **软删 ≠ 物理删除**：数据物理保留，只是从视图隐藏。物理清理过期归档数据另起独立任务，不混进归档动作。

## 1. 分层结构

沿用现有目录约定，不引入新分层：

```
com.fundpilot.backend
├── common/                    # 已有：AbstractEntity / ApiResponse / EnumValue
├── exception/                 # 新增：GlobalExceptionHandler + 业务异常
├── fund/
│   ├── entity/                # 扩展：UserFundStrategyEntity；新增：SignalLogEntity；改造：FundFlowEntity → FundTransactionEntity
│   ├── enums/                 # 新增：FundCategory / StrategyParamStatus / StrategyOperationType / SignalType；改造：FundStatus / FundFlowSource → FundTransactionSource / FundFlowStatus → FundTransactionStatus
│   ├── repository/            # 新增：4 个 JpaRepository
│   ├── dto/                   # 新增：策略配置 / 信号视图 / 操作确认
│   ├── service/
│   │   ├── DisciplineStrategyService   # 信号引擎：每日生成 + 操作确认
│   │   ├── StrategyConfigService       # 参数 CRUD + 状态机
│   │   ├── FundPositionService         # 事实仓位聚合（读 FundTransaction，仅 CONFIRMED）
│   │   ├── SignalGenerationJob         # 每日 14:50 定时任务入口
│   │   ├── NavConfirmJob               # 每晚净值公布后回填 PENDING_NAV 交易
│   │   └── support/                    # 纯函数工具：档位默认表、系数表、clamp
│   └── controller/            # 扩展 FundController；可拆 StrategyController
└── config/                    # 已有：JPA 审计等
```

> 不引入 Hexagonal / DDD 风格的 domain/application/infrastructure 分层。项目体量还撑不起，强行做反而干扰阅读。

## 2. 实体设计

### 2.1 `FundEntity`（基金产品）

只描述"是哪只基金" + 跟策略无关的**基金属性和市场事实 / 持有事实**，不存仓位事实。

| 字段                   | 类型                        | 说明                                                                |
|----------------------|---------------------------|-------------------------------------------------------------------|
| 继承自 AbstractEntity   |                           | id / version / 审计 / 软删                                            |
| fundCode             | String                    |                                                                   |
| fundName             | String                    |                                                                   |
| investmentTarget     | InvestmentTarget enum     | 已有                                                                |
| operationMode        | OperationMode enum        | 已有                                                                |
| investmentPhilosophy | InvestmentPhilosophy enum | 已有                                                                |
| status               | **FundStatus**（重做）        | 未建仓 / 持仓中 / 已清仓                                                   |
| plannedTotalAmount   | BigDecimal, nullable      | 计划总仓位（纪律意图）；建仓时（NOT_OPENED → HOLDING）必须填，HOLDING/CLOSED 状态下非 null |
| peakNav              | BigDecimal                | 前高净值（基金历史最高累计净值，市场事实，每月更新）                                        |
| holdingPeriodPeakNav | BigDecimal, nullable      | 持有期最高净值（我买入后的最高净值，算移动止盈用）；未建仓时为 null                              |
| openedAt             | Instant, nullable         | 首次建仓时间（持有事实，跟策略版本无关）；`CLOSED → NOT_OPENED` 时清空                    |

**移除**：现有的 `amount` 和 `shares` 字段。事实仓位由 `FundTransactionEntity` 聚合得出，不在产品实体上冗余。

> `plannedTotalAmount` / `peakNav` / `holdingPeriodPeakNav` / `openedAt` 都从 `UserFundStrategyEntity`
> 搬过来——它们跟策略参数版本无关，是基金本身的属性、市场事实或持有事实。换参数版本不影响这些值。

> **`FundStatus` 变更不走日志表**：变更不频繁（一只基金一辈子也就 NOT_OPENED → HOLDING → CLOSED → 偶尔回
> NOT_OPENED），没有审计/复盘需求，当前状态字段足够。如果以后真需要审计，再补日志表不迟。

### 2.2 `UserFundStrategyEntity`（策略参数 + 该版本下的运行时状态）

一份记录 = 一份策略快照。改参数 = 新建一份 DRAFT，激活时旧的转 RETIRED。

只放跟**策略参数版本相关**的字段。基金属性（`plannedTotalAmount`、`peakNav`、`holdingPeriodPeakNav`、`openedAt`）和全局硬约束（
`singlePositionLimit`）不在这里。

| 字段                            | 类型                        | 说明                                                                       |
|-------------------------------|---------------------------|--------------------------------------------------------------------------|
| 继承自 AbstractEntity            |                           |                                                                          |
| fund                          | ManyToOne FundEntity LAZY |                                                                          |
| userId                        | Long, nullable            | 预留，本期不用                                                                  |
| paramStatus                   | StrategyParamStatus enum  | 待校准 / 已校准 / 已生效 / 已退役                                                    |
| fundCategory                  | FundCategory enum         | 宽基 / 行业 / 主动 / 混合                                                        |
| tier1Drawdown ~ tier4Drawdown | BigDecimal                | 四档回撤阈值（百分比，如 0.08 = 8%）                                                  |
| tier1Ratio ~ tier4Ratio       | BigDecimal                | 四档加仓比例（占 plannedTotalAmount 的百分比，如 0.15 = 15%）；四档之和 ≤ 90%                |
| weeklyCoolDownThreshold       | BigDecimal                | 单周跌幅冷静阈值（百分比，如 0.10 = 10%），用于强制冷静                                        |
| stopLossPullbackPercent       | BigDecimal                | 移动止盈回落幅度（百分比，如 0.08 = 8%）                                                |
| tier1AddedAt ~ tier4AddedAt   | Instant, nullable         | 方案B 四档**实际执行**时间戳，null = 未加。**只在用户通过 POST /operations 确认执行时写入**，信号生成只读不写 |

> **不在主表存 `activatedAt` / `retiredAt`**：走激活表 `FundStrategyActivationEntity`（见 2.7），主表只保留当前状态
`paramStatus`。
> **不在主表存 `openedAt`**：搬到 `FundEntity`（持有事实，跟策略版本无关）。

**tier1~4AddedAt 写入时机**（重要）：

- 信号生成（`generateDailySignals`）只读不写 `tierNAddedAt`（例外：反弹清空是状态机自动行为，随信号生成一起写）
- `POST /operations` 确认执行时才写
- operationType = `ADD_TIER_1` 写 `tier1AddedAt`，依次类推
- 用户不回应 → `tierNAddedAt` 仍是 null → 下次还在该档会再提示

**唯一性约束**：同一 `fund_id` 在未软删的记录里，最多 1 份 `paramStatus = ACTIVE`。Postgres 部分唯一索引：

```sql
CREATE UNIQUE INDEX uq_user_fund_strategy_active
    ON user_fund_strategy (fund_id) WHERE param_status = 'ACTIVE' AND deleted_date IS NULL;
```

### 2.3 `FundTransactionEntity`（基金交易 · 改名自 `FundFlowEntity`）

原 `FundFlowEntity` 改名为 `FundTransactionEntity`，承载**基金交易事件**——买入、卖出、转入、转出。合并了原 `StrategyOperationEntity` 的策略上下文字段（`signalLog` / `operationType`），手动操作和转账这些字段为 null。

**字段**

| 字段                    | 类型                                  | 说明                                                                       |
|-----------------------|-------------------------------------|--------------------------------------------------------------------------|
| 继承自 AbstractEntity    |                                     |                                                                          |
| fund                  | ManyToOne FundEntity LAZY           |                                                                          |
| source                | FundTransactionSource enum          | BUY / SELL / TRANSFER_IN / TRANSFER_OUT                                  |
| status                | FundTransactionStatus enum          | PENDING_NAV / CONFIRMED / CANCELLED                                      |
| shares                | BigDecimal, nullable                | 买入下单时 null（等净值回填）；卖出下单时已知；转账已知；回填后永不改                                      |
| amount                | BigDecimal, nullable                | 买入下单时已知；卖出下单时 null（等净值回填）；转账已知；回填后永不改                                      |
| nav                   | BigDecimal, nullable                | 成交净值，净值确认后回填；下单时 null                                                      |
| confirmTime           | Instant, nullable                   | 净值确认时间（区别于 `createdDate` 下单时间）                                            |
| signalLog             | ManyToOne SignalLogEntity LAZY, nullable | 由信号触发的交易才填；手动操作 / 转账 null                                                |
| operationType         | StrategyOperationType, nullable     | 由信号触发的交易才填（OPEN / ADD_TIER_N / TRAILING_STOP / ...）；手动 / 转账 null            |
| relatedTransaction    | OneToOne FundTransactionEntity LAZY, nullable | 基金转换时指向另一条腿（A 转 OUT + B 转 IN 互指）；其他场景 null                              |

> `FundTransactionEntity` 合并了原 `StrategyOperationEntity` 的职责。一次买入/卖出信号确认 = 写一条 FundTransaction（含 `signalLog` + `operationType`）；`OVERRIDE_SKIP` 不写交易（用户表态跳过 = 无交易记录，与"被动忽略"在数据上不区分）；转账 / 手动操作不带策略上下文字段。

**交易时序与状态机**

```
PENDING_NAV  ── NavConfirmJob 跑完 ──→  CONFIRMED    (买入：回填 shares=amount/nav；卖出：回填 amount=shares×nav)
PENDING_NAV  ── 用户撤单 ──→           CANCELLED
```

- 下单那一刻写入已知侧：买入 `amount` 已知、`shares=null`；卖出 `shares` 已知、`amount=null`；`nav=null`、`status=PENDING_NAV`
- 当晚基金公司公布当日净值，`NavConfirmJob` 跑：查 `FundNavHistoryEntity` 拿当日净值，回填另一侧 + `nav` + `confirmTime`，转 `CONFIRMED`
- 回填后字段不再改变（符合事实账目"回填后永不改"原则）

**持仓计算**

```
持仓份额 = Σ( shares × direction ) WHERE status = 'CONFIRMED'     # direction: BUY/TRANSFER_IN = +1, SELL/TRANSFER_OUT = -1
在途订单 = Σ( ... ) WHERE status = 'PENDING_NAV'                  # 前端单独展示"已下单待确认"
持仓金额 = 持仓份额 × 当前净值                                       # 实时算，不存
持仓占比 = 持仓金额 / 总权益仓位
```

> 今天净值上涨导致"钱变多了"——不需要任何交易记录，只是 `持仓份额 × 新净值` 自然变大。只有真正发生买卖时才写交易。

**转换（基金 A → 基金 B）**

两条交易通过 `relatedTransaction` 互指：

```
FundTransaction(fund=A, source=TRANSFER_OUT, shares=500, ...)
FundTransaction(fund=B, source=TRANSFER_IN,  shares=500, ...)
  ↑ 两条 relatedTransaction 指向对方
```

撤单时两条一起 CANCELLED。

### 2.4 `SignalLogEntity`（每日信号快照 · 新增）

每日 14:50 定时任务对每只绑定 `ACTIVE` 策略的基金跑一遍信号引擎，把当天的信号结果**冻进一张快照表**。信号是当时那一刻的市场切片，14:50 的指标/净值收盘后无法复算，必须当时落库。

| 字段                      | 类型                        | 说明                                            |
|-------------------------|---------------------------|-----------------------------------------------|
| 继承自 AbstractEntity      |                           |                                               |
| fund                    | ManyToOne FundEntity LAZY |                                               |
| userFundStrategy        | ManyToOne LAZY            | 当天生效的参数版本                                     |
| signalDate              | LocalDate                 | 信号日期，14:50 那个交易日                              |
| signalType              | SignalType enum           | NONE / BUILD / ADD / SELL                     |
| triggerNav             | BigDecimal, nullable      | 触发时净值（signalType ≠ NONE 时）                    |
| triggerTier             | Integer, nullable         | 触发档位 1~4（ADD 时）                               |
| coefficient             | BigDecimal, nullable      | 最终系数（ADD 时）                                   |
| suggestedMeasure        | `Measure` @Embedded, nullable | 系统建议量 + 单位（见下方值对象定义）：BUILD/ADD 时 `{value, AMOUNT}`（建议金额，买入按金额下单）；SELL 时 `{value, SHARES}`（建议份额，卖出按份额下单）；NONE 时为 null |
| reason                  | String, nullable          | NO_ACTIVE_STRATEGY / TIER_HIT / COOLDOWN / WATCH_AND_WAIT / ... |
| warnings                | JSONB / TEXT, nullable    | 强提示列表（强制冷静、破位观望等）                             |
| hardConstraintBreaches  | JSONB / TEXT, nullable    | 命中硬约束被截断的说明                                   |
| generatedAt             | Instant                   | 任务执行时间                                        |

**唯一性约束**：`(fund_id, signal_date)` 在未软删的记录里唯一——每只基金每个交易日最多一行（含 NONE）。

```sql
CREATE UNIQUE INDEX uq_signal_log_daily
    ON signal_log (fund_id, signal_date) WHERE deleted_date IS NULL;
```

> 每日一行/基金（含"今天无信号"的 NONE 行），量级和 `fund_nav_history` 同级，兼作"系统每天评估过这只基金"的审计轨迹。
>
> **被动忽略可追踪**：`signalType ≠ NONE` 且无对应 `FundTransactionEntity` 的 SignalLog 行 = 系统提示了但用户没回应（含主动 SKIP 和被动忽略，两者在数据上不区分）。`LEFT JOIN fund_transaction ON signal_log.id = fund_transaction.signal_log_id WHERE transaction.id IS NULL AND signal_type <> 'NONE'` 直接列出。
>
> **`suggestedMeasure` 是带单位的量值**：值对象 `Measure { BigDecimal value; MeasureUnit unit }`，用 `@Embedded` 映射成两列 `suggested_value` + `suggested_unit`。由 `unit` 决定 `value` 的语义，`BUILD`/`ADD` 时 `unit=AMOUNT`、`value` 是建议金额（买入按金额下单），`SELL` 时 `unit=SHARES`、`value` 是建议份额（卖出按份额下单）。另一侧（买入的份额、卖出的金额）可由 `triggerNav` 实时算出，不冗余存。

#### 值对象 `Measure`（新增 · `@Embeddable`）

通用的"带单位量值"抽象——一个数值配它的量纲单位。本期只在 `SignalLogEntity.suggestedMeasure` 使用；如果以后别处有"值 + 单位"二选一的场景可复用同类型。

```
@Embeddable
class Measure {
  BigDecimal value;
  MeasureUnit unit;
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `value` | BigDecimal | 量的数值 |
| `unit` | `MeasureUnit` | 量纲：`AMOUNT`（金额，元）/ `SHARES`（份额，份） |

> **操作表已合并进 `FundTransactionEntity`**：原 `StrategyOperationEntity` 的策略上下文字段（`signalLog` / `operationType`）落到 `FundTransactionEntity` 上作为 nullable 字段。用户回应信号 = 写一条带策略上下文的 FundTransaction；`OVERRIDE_SKIP` 不写交易（跳过即无记录，与被动忽略在数据上不区分）；手动操作 / 转账的策略上下文字段为 null。详见 2.3。

### 2.5 `StrategyBacktestEntity`（回测结果 · 新增）

`CALIBRATED → ACTIVE` 的硬性前置：必须有一份**有效**的回测结果跑赢三条基准线。

| 字段                        | 类型             | 说明                       |
|---------------------------|----------------|--------------------------|
| 继承自 AbstractEntity        |                |                          |
| userFundStrategy          | ManyToOne LAZY | 被回测的那一版参数                |
| backtestStart             | LocalDate      | 回测起点                     |
| backtestEnd               | LocalDate      | 回测终点                     |
| strategyReturn            | BigDecimal     | 策略最终收益率                  |
| strategyMaxDrawdown       | BigDecimal     | 策略期间最大回撤                 |
| benchmarkHs300Return      | BigDecimal     | 沪深300 同期收益率              |
| benchmarkAllInReturn      | BigDecimal     | 一次性 all-in 同期收益率         |
| benchmarkDcaReturn        | BigDecimal     | 等额定投同期收益率                |
| benchmarkHs300MaxDrawdown | BigDecimal     | 沪深300 同期最大回撤             |
| benchmarkAllInMaxDrawdown | BigDecimal     | 一次性 all-in 同期最大回撤        |
| benchmarkDcaMaxDrawdown   | BigDecimal     | 等额定投同期最大回撤               |
| passed                    | boolean        | 是否通过（收益跑赢三条且回撤 ≤ all-in） |

> 默认回测窗口"近一年"，但允许人工指定起止日期，便于跨周期对比。

> 回测结果与参数版本一对多：同一版参数可以多次回测（窗口不同/数据更新），但只要有**任意一份** `passed = true`
> 的结果，就允许激活。

### 2.6 `FundNavHistoryEntity`（基金净值历史 · 新增）

展示和回测都依赖"最近一次确认净值"和历史净值序列，必须有一张表承载。本期由人工/调用方录入，预留未来对接行情接口。

| 字段                 | 类型                        | 说明                        |
|--------------------|---------------------------|---------------------------|
| 继承自 AbstractEntity |                           |                           |
| fund               | ManyToOne FundEntity LAZY |                           |
| navDate            | LocalDate                 | 净值日期，业务唯一键                |
| nav                | BigDecimal                | 单位净值                      |
| accumulatedNav     | BigDecimal                | 累计净值（复权），算前高用             |
| source             | NavSource enum            | `MANUAL` / `API_SYNC`（预留） |

**唯一性约束**：`(fund_id, nav_date)` 在未软删的记录里唯一。

```sql
CREATE UNIQUE INDEX uq_fund_nav_history
    ON fund_nav_history (fund_id, nav_date) WHERE deleted_date IS NULL;
```

> 接东方财富/天天基金等数据源时，只是新增一条 `source = API_SYNC` 的写入路径，业务代码全不动。

### 2.7 `FundStrategyActivationEntity`（基金策略激活记录 · 新增）

记录**基金启用了哪版参数、用了多久**——基金维度的激活谱系，不是参数内部状态机。一行 = 基金的一次激活任期（某参数版本从启用到退役）。

| 字段                 | 类型                       | 说明                   |
|--------------------|--------------------------|----------------------|
| 继承自 AbstractEntity |                          |                      |
| fund               | ManyToOne LAZY           | 基金                   |
| userFundStrategy   | ManyToOne LAZY           | 被启用的那版参数             |
| activatedAt        | Instant                  | 启用时间                 |
| retiredAt          | Instant, nullable        | 退役时间；null = 当前生效     |
| supersededBy       | ManyToOne LAZY, nullable | 顶替它的新版本（退役时填，主动退役不填） |

- **激活**：插一行（`retiredAt = null`）。
- **退役**：回填 `retiredAt`；被新版本顶替时同步填 `supersededBy`。
- **查当前生效**：`retiredAt IS NULL`（同一基金最多 1 条，由主表 ACTIVE 唯一约束保证一致）。
- **查激活历史**：按 `fund` + `activatedAt` 排序。

> `StrategyConfigService` 的 `activate` / `retire` 方法里统一处理激活记录的写入与回填，业务层不需要手动调。
>
> **只记启用/退役，不记参数内部状态变化**：`DRAFT→CALIBRATED`、`DRAFT→RETIRED` 等参数内部状态在主表 `paramStatus`
> 里够用，没有审计价值，不进激活表。
>
> `FundStatus` 变更**不**走激活表：变更不频繁，无审计需求，当前状态字段足够。如果以后真需要审计，再补对称的
`FundStatusChangeEntity` 不迟。

## 2A. 激活流程（重点）

```
DRAFT
  │
  │  人工确认参数定好
  ▼
CALIBRATED  ── (尚未回测)
  │
  │  POST /strategies/{id}/backtest  (传回测窗口/净值序列)
  │  → 系统按该版本参数在历史净值上模拟执行
  │  → 落 StrategyBacktestEntity 一行
  │
  │  POST /strategies/{id}/activate
  │  → 服务先查：是否存在 passed = true 的回测结果？
  │     否 → 抛 BusinessException("NO_VALID_BACKTEST")，不激活
  │     是 → CALIBRATED → ACTIVE
  │        同基金旧 ACTIVE → RETIRED
  │
  ▼
ACTIVE
  │
  │  新版参数被激活 或 人工停用
  ▼
RETIRED
```

**"通过"判定（`passed`）**：在相同回测窗口下，策略须**同时**满足：

1. 最终收益率大于三条基准线收益率（沪深300 同期、一次性 all-in、等额定投）
2. 策略最大回撤 ≤ all-in 同期最大回撤

两条都成立才算通过，`passed = true`。任一不成立 → `passed = false`，不允许激活。

> 收益率条件是策略的收益价值证明；回撤条件是纪律策略的设计底线——分档加仓的回撤不该比一次性 all-in 还大，否则逻辑破产。回撤用
`≤` 而非 `<`，避免临界值抖动导致反复激活失败。不要求回撤跑赢三条基准（定投/沪深300 回撤可能很小，门槛过高会让回测前置形同虚设）。

## 3. 枚举

### 3.1 `FundStatus`（改造现有）

```
NOT_OPENED   未建仓
HOLDING      持仓中
CLOSED       已清仓
```

合法跃迁：

- `NOT_OPENED → HOLDING`（建仓首笔确认）
- `HOLDING → CLOSED`（持仓份额归零）
- `CLOSED → NOT_OPENED`（人工确认继续观察，清空持有期相关的运行时状态）

> `ARCHIVED` 不作为 `FundStatus` 的值。归档通过软删除表达（见 0 节"硬性原则"和《落地讨论》1.1A）。
> `canTransitionTo` 内置在枚举里。

### 3.2 `StrategyParamStatus`（新增）

```
DRAFT        待校准
CALIBRATED   已校准
ACTIVE       已生效
RETIRED      已退役
```

### 3.3 `FundCategory`（新增）

```
BROAD_BASE   宽基
SECTOR       行业
ACTIVE_FUND  主动
MIXED        混合
```

> 注意 `ACTIVE_FUND` 不叫 `ACTIVE`，避开与 `StrategyParamStatus.ACTIVE` 同名歧义。

### 3.4 `StrategyOperationType`（新增 · 落到 `FundTransactionEntity.operationType`）

```
OPEN                建仓
ADD_TIER_1 ~ 4      加仓（四档）
TRAILING_STOP       移动止盈
LOGIC_STOP_LOSS     逻辑破坏止损
REBALANCE           再平衡减仓
OVERRIDE_ADD        强提示下仍加仓（userActualAmount ≠ SignalLog.suggestedMeasure.value 的留痕）
```

> `OVERRIDE_SKIP` 不在枚举里——用户主动跳过信号不写交易记录，与被动忽略在数据上不区分（见 2.3 末尾"被动忽略可追踪"）。要区分的话以后在 `SignalLogEntity` 加 `responseType` 字段，本期不做。

### 3.5 `SignalType`（新增）

```
NONE    当天无信号（已评估，未触发任何动作建议）
BUILD   建仓信号
ADD     加仓信号（具体档位看 SignalLog.triggerTier）
SELL    卖出信号（具体类型：移动止盈 / 逻辑止损 / 再平衡，由操作确认时落到 operationType）
```

> `NONE` 不是"未评估"，而是"评估过了但今天不该动"——和"未生成"区分。每日定时任务对每只 `ACTIVE` 基金都生成一行，含 NONE 行，作为审计轨迹。

### 3.6 `NavSource`（新增）

```
MANUAL      人工录入
API_SYNC    接口同步（预留，本期不实现）
```

### 3.7 `FundTransactionSource`（改造 · 原 `FundFlowSource` 改名 + 收敛）

```
BUY            买入（份额增加；金额已知、份额等净值回填）
SELL           卖出（份额减少；份额已知、金额等净值回填）
TRANSFER_IN    转入（份额增加；基金转换的入账腿或外部转入）
TRANSFER_OUT   转出（份额减少；基金转换的出账腿或外部转出）
```

> 原 `FundFlowSource` 的 `INCREASE` / `DECREASE` 语义太泛，收敛成明确的 `BUY` / `SELL`；`INVEST`（定投扣款）本期不做定投，去掉——定投每期就是一笔普通 `BUY`，要做定投时每期落一条 `BUY`，不单独设类型（定投计划是另一张 `DcaPlanEntity` 表的事，和交易来源无关）；转换用 `TRANSFER_IN` + `TRANSFER_OUT` 两条交易通过 `relatedTransaction` 互指表达。

### 3.8 `FundTransactionStatus`（改造 · 原 `FundFlowStatus` 改名 + 重做值）

```
PENDING_NAV    已下单，等净值确认（买入 amount 已知 / 卖出 shares 已知，另一侧 null）
CONFIRMED      净值确认完成（shares + amount + nav 都已回填）
CANCELLED      撤单（用户撤单；转换撤单时两条腿一起 CANCELLED）
```

> 原 `FundFlowStatus` 的值整体替换。`PENDING_NAV` 是基金交易的正常中间态——买入下单时知道金额、不知道份额（净值未公布），卖出下单时知道份额、不知道金额。`NavConfirmJob` 每晚净值公布后跑，把 `PENDING_NAV` 行回填另一侧并转 `CONFIRMED`。聚合只看 `CONFIRMED` 行。

### 3.9 `MeasureUnit`（新增 · 值对象 `Measure` 的量纲枚举）

```
AMOUNT   金额（元）
SHARES   份额（份）
```

> 配合 `Measure` 值对象使用（见 2.4 末尾值对象定义）。`SignalLog.suggestedMeasure.unit` 决定 `value` 是金额还是份额。

## 4. Service 层职责切分

### 4.1 `StrategyConfigService`

- `createDraft(fundId, configRequest)` → 写一条 `DRAFT`
- `updateDraft(strategyId, configRequest)`
- `calibrate(strategyId)` → `DRAFT → CALIBRATED`
- `backtest(strategyId, BacktestRequest)` → 用历史净值跑回测，结果包含三条基准线对比
- `activate(strategyId)` → 校验该版本已通过回测且跑赢三条基准线，通过后 `CALIBRATED → ACTIVE`，同基金旧 `ACTIVE` 转
  `RETIRED`
- `retire(strategyId)` → 主动停用
- 查询：`findActive(fundId)`、`listByFund(fundId)`、`getBacktestResult(strategyId)`

> 激活的硬性前置条件：参数必须先通过回测验证，跑赢三条基准线（沪深300、一次性 all-in、等额定投）才允许 `activate`
> 。回测结果落表，便于追溯和复跑。
>
> **激活/退役写激活表**：`activate` 写一行 `FundStrategyActivationEntity`（`retiredAt = null`），`retire` 回填 `retiredAt` 和
`supersededBy`。`calibrate` 不写激活表（参数内部状态变化不记）。业务层不需要手动调。
>
> **`CLOSED → NOT_OPENED` 时的参数处理**：清仓后人工确认重新观察，基金回 `NOT_OPENED` 时，原 `ACTIVE` 参数强制退到
`CALIBRATED`（回填激活记录的 `retiredAt`）。重新激活前必须重新回测。

### 4.2 `FundPositionService`

按以下规则给出某只基金当前的"事实仓位"，第一版实时聚合不缓存：

- **持仓份额** = `Σ FundTransactionEntity.shares × direction`（`status = CONFIRMED` 的交易；`PENDING_NAV` 算在途，不进持仓）
- **在途订单** = `Σ ... WHERE status = 'PENDING_NAV'`（前端单独展示"已下单待净值确认"）
- **持仓金额** = 持仓份额 × 最新净值（净值由调用方传入，本期不接行情源）
- **持仓占比** = 持仓金额 / 总权益仓位
- **持仓成本** = `Σ amount × direction`（CONFIRMED 状态；不算盈亏，只是名义投入）

direction: `BUY`/`TRANSFER_IN` = +1，`SELL`/`TRANSFER_OUT` = -1。不存任何中间结果。性能成问题时再加快照表。

### 4.3 `DisciplineStrategyService`

信号引擎。两类入口分开：**生成**（定时任务调）和**确认**（用户调）。信号引擎核心算法是无状态纯函数，两类入口都复用它。

```
generateDailySignals(signalDate)                     # 定时任务入口：14:50 跑，遍历所有 ACTIVE 基金
  → 对每只基金调 evaluateSignal(fund, strategy, marketIndicators, capitalContext)
  → 落一行 SignalLogEntity（含 NONE）
confirmOperation(signalLogId, confirmRequest)        # 用户回应入口：写 FundTransaction + 推进状态（OVERRIDE_SKIP 不写交易）
```

**`evaluateSignal` 的固定步骤**（纯函数，输入 `FundEntity` + `UserFundStrategyEntity` + 市场指标 + 资金上下文，输出信号结果）：

```
1. 校验 FundStatus 是否允许出该类信号（见组合表）；不允许 → 返回 NONE
2. 校验 ACTIVE 策略存在；没有 → 返回 NONE，reason=NO_ACTIVE_STRATEGY
3. 计算当前回撤（基于 FundEntity.peakNav 和最近确认净值）
4. 反弹清空判断（加仓信号专属）：当前档位比已加最深档浅 → 清空更深档的 tierNAddedAt（状态机自动行为，随生成一起写）
5. 计算业务信号（档位 / 系数 / 强提示 / 卖出类型）
6. 跑硬约束截断
7. 组装结果：signalType、reason、triggerTier、coefficient、suggestedMeasure（Measure 值对象）、warnings[]、hardConstraintBreaches[]
```

**`generateDailySignals` 的职责**：

- 遍历所有"绑定 ACTIVE 策略 + 未软删"的基金
- 通过 `MarketIndicatorProvider` 拉每只基金当天的市场指标（本期可半自动）
- 通过 `FundPositionService` 算每只基金的资金上下文（用于硬约束）
- 调 `evaluateSignal`，结果落 `SignalLogEntity`（一行/基金/日，含 NONE）
- 唯一性约束兜底重复执行：同一 `(fund_id, signal_date)` 已存在则跳过或覆盖（按业务定，倾向覆盖以支持重跑）

**`confirmOperation` 的推进逻辑**（按 operationType switch）：

| operationType                                     | 推进动作                                                                                                |
|---------------------------------------------------|-----------------------------------------------------------------------------------------------------|
| `OPEN`                                            | 写 FundTransaction（`source=BUY`，`amount` 已知，`shares=null`，`status=PENDING_NAV`）；FundEntity.status = HOLDING；FundEntity.openedAt = now；初始化 holdingPeriodPeakNav |
| `ADD_TIER_1 ~ 4`                                  | 写 FundTransaction（`source=BUY`，同上）；UserFundStrategy.tierNAddedAt = now                                 |
| `TRAILING_STOP` / `LOGIC_STOP_LOSS` / `REBALANCE` | 写 FundTransaction（`source=SELL`，`shares` 已知，`amount=null`，`status=PENDING_NAV`）；如果聚合后 CONFIRMED 持仓份额归零 → FundEntity.status = CLOSED |
| `OVERRIDE_ADD`                                    | 同对应 `ADD_TIER_N`，`amount` 来自请求（可与 `SignalLog.suggestedMeasure.value` 不同，差值本身就是 override 留痕，无需专门字段）            |
| 用户主动跳过 / 不回应                                    | 不写 FundTransaction，不推进任何状态（被动忽略与主动跳过在数据上不区分）                                                          |

> 买入写 `amount` 不写 `shares`，卖出写 `shares` 不写 `amount`——另一侧等 `NavConfirmJob` 当晚净值确认后回填，状态从 `PENDING_NAV` 转 `CONFIRMED`（见 4.5）。
>
> 建议侧字段（triggerNav / triggerTier / coefficient / suggestedMeasure）已在 `SignalLogEntity` 里冻好，`FundTransaction` 通过 `signalLog` 关联即可拿到，不在交易表里冗余。`ConfirmOperationRequest` 因此不用带 `MarketIndicator` / `CapitalContext`，确认时无需复算信号。

**关键约束：查询接口只读、生成任务才写**

- `evaluateSignal` 是纯函数，本身不读写数据库
- `SignalLogEntity` 的写入只发生在 `generateDailySignals` 里（含步骤 4 的反弹清空对 `tierNAddedAt` 的写——它是状态机自动行为，跟着生成走）
- 档位"已加"标记（`tierNAddedAt`）的另一种写入发生在 `confirmOperation` 里
- 查询接口（`GET /signals/...`、`GET /signals/today` 等）只读 `SignalLogEntity`，不触发任何计算或写入
- 加仓建议金额 = `plannedTotalAmount × tierNRatio × 最终系数`，再被硬约束截断（在 `evaluateSignal` 里算）

### 4.4 `NavConfirmJob`

每晚基金公司公布当日净值后跑（一般 20:00-21:00），把当天 `PENDING_NAV` 的 `FundTransactionEntity` 行回填齐并转 `CONFIRMED`。

```
run()
  1. 查所有 status = PENDING_NAV 且 fund 今日有 NavHistory 行的交易
  2. 对每条：
     - 查 FundNavHistoryEntity 拿当日 nav
     - source = BUY  → shares = amount / nav
     - source = SELL → amount = shares × nav
     - 填 nav、confirmTime = now，status = CONFIRMED
  3. 撤单窗口外的 PENDING_NAV 行保留（基金公司未公布净值的边缘情况），等次日 job 再处理
```

> 用户撤单走另一个路径：`POST /api/transactions/{id}/cancel` 把 `PENDING_NAV` 行转 `CANCELLED`（基金转换撤单时两条 `relatedTransaction` 一起 CANCELLED）。`CONFIRMED` 行不可撤。

### 4.5 `support/` 纯函数工具

- `DefaultTierTable`：宽基/行业/主动/混合 × 四档的**默认回撤阈值 + 默认加仓比例**表（建仓 DRAFT
  时填默认值用）。回撤阈值取自框架第四节，四档加仓比例四类基金统一用金字塔递增（深档加更多），合计 90%：

  | 类型 | 一档回撤 | 二档回撤 | 三档回撤 | 四档回撤 | 一档比例 | 二档比例 | 三档比例 | 四档比例 |
    |---|---|---|---|---|---|---|---|---|
  | 宽基 | -8% | -15% | -25% | -35% | 15% | 20% | 25% | 30% |
  | 行业 | -15% | -25% | -35% | -45% | 15% | 20% | 25% | 30% |
  | 主动 | -12% | -20% | -30% | -40% | 15% | 20% | 25% | 30% |
  | 混合 | -12% | -22% | -32% | -40% | 15% | 20% | 25% | 30% |

  > 建仓首笔 10% + 四档 90% = 100% 计划总仓位，正好用满。差异化体现在回撤阈值上，比例不分类型调。

- `DefaultCoolDownTable`：单周跌幅冷静阈值默认值，按 `FundCategory`：宽基 8% / 行业 12% / 主动 10% / 混合 10%
- `CoefficientTable`：年线/MACD/成交量 → 系数的查表
- `CoefficientCombiner`：合成 + clamp(0.3, 1.5)
- `HardConstraintConfig`：**全局硬约束规则**（代码常量），跟框架第九节一致：
    - `singlePositionLimit`（按 FundCategory）：宽基/主动/混合 20%，行业 15%
    - `categoryPositionLimit`：单类型上限 30%（同 FundCategory 汇总）
    - `totalEquityPositionLimit`：总权益仓位上限 80%
    - `singleAddRatioLimit`：单次加仓 ≤ 该品种剩余可用仓位的 50%
    - `minHoldDays`：建仓后 7 天内不赎回（逻辑破坏止损除外）
- `HardConstraintChecker`：五条硬约束统一入口，调用 `HardConstraintConfig` 读上限

放静态工具类或 `@Component` 都可以，倾向静态——它们都是纯函数。

## 5. DTO 设计

### 5.1 请求 DTO（关键的几条）

`StrategyConfigRequest`

- 5 个参数字段；`fundCategory`；`plannedTotalAmount`

`MarketIndicators`（信号生成入参，由 `MarketIndicatorProvider` 在 14:50 拉取；本期可半自动灌入）

- `currentNav`、`peakNav`（可空，传则覆盖记录中的值）
- `priceAboveYearLine` / `yearLineRising`（推年线状态）
- `weeklyMacdState`（enum：底背离 / 绿柱缩小 / 红柱缩小 / 绿柱扩大）
- `volumeState`（enum：地量 / 正常 / 放量下跌）
- `weeklyDropPercent`
- `isSixtyDayHigh`（仅建仓用）

`CapitalContext`

- `totalInvestableCapital`、`currentTotalEquityPosition`、`currentCategoryPosition`（用于跑硬约束）

`ConfirmOperationRequest`

- `signalLogId`（必填，指向被回应的那条 `SignalLogEntity`）
- `operationType`（必填，决定写哪种 `FundTransaction`）
- `actualAmount`（买入类操作必填：`OPEN` / `ADD_TIER_N` / `OVERRIDE_ADD`；买入下单时知道金额、不知道份额）
- `actualShares`（卖出类操作必填：`TRAILING_STOP` / `LOGIC_STOP_LOSS` / `REBALANCE`；卖出下单时知道份额、不知道金额）

> 买入传金额、卖出传份额——对应基金交易的现实时序，用户下单时确实知道哪一侧。另一侧等 `NavConfirmJob` 当晚净值确认后回填。
>
> 不再需要在请求里带 `MarketIndicators` / `CapitalContext` / `actualNav` / `overrideReason`——建议侧字段已在 `SignalLogEntity` 里冻好，确认时直接通过 `signalLogId` 关联读取，不复算信号；`actualNav` 在下单时根本拿不到（净值未公布）；`overrideReason` 已删除（`OVERRIDE_ADD` 的留痕靠 `FundTransaction.amount` vs `SignalLog.suggestedMeasure.value` 的差值，不需要专门字段）。
>
> 用户主动跳过信号（`OVERRIDE_SKIP`）不调本接口、不写交易——与"被动忽略"在数据上不区分。

### 5.2 响应 DTO

信号视图直接对应 `SignalLogEntity` 行（查询接口返回），不再有"算一次返回一次"的临时响应：

```
SignalView {
  signalLogId: Long
  fundId: Long
  signalDate: LocalDate
  signalType: SignalType        // NONE / BUILD / ADD / SELL
  reason: String?               // NO_ACTIVE_STRATEGY / TIER_HIT / COOLDOWN / WATCH_AND_WAIT / ...
  triggerNav: BigDecimal?
  triggerTier: Integer?         // ADD 时
  coefficient: BigDecimal?
  suggestedMeasure: Measure?   // 系统建议量 + 单位；BUILD/ADD 时 {value, AMOUNT}（建议金额），SELL 时 {value, SHARES}（建议份额）；NONE 时为 null
  warnings: List<Warning>       // 强提示：强制冷静、破位观望等
  hardConstraintBreaches: List<Breach>  // 命中硬约束被截断的说明
  respondedAt: Instant?         // 已回应时指向 FundTransaction.confirmTime（或 createdDate，下单时刻）；null = 未回应（被动忽略或主动跳过，两者不区分）
}

- **纯函数工具**（`CoefficientCombiner`、`HardConstraintChecker`、`DefaultTierTable`）：纯单测，覆盖边界（系数 0.3 /
  1.5、各档位边界、单周冷静临界）。

- **信号引擎**（`evaluateSignal` 纯函数 + `generateDailySignals` 任务）：用 mock 仓位 + 构造好的 `UserFundStrategyEntity`，跑场景：
    - 无 ACTIVE 参数 → `signalType=NONE`，`reason=NO_ACTIVE_STRATEGY`
    - 进二档第一次 → `signalType=ADD`，`triggerTier=2`，出建议金额
    - 进二档已加 → `signalType=NONE`（已加档不重复提示）
    - 反弹清空场景（`tierNAddedAt` 被清）
    - 破位观望强提示（`warnings` 非空）
    - 五条硬约束分别命中（`hardConstraintBreaches` 非空、`suggestedMeasure.value` 被截断）
    - `generateDailySignals` 任务：多只基金批量生成、重复执行幂等（同日重跑覆盖或跳过）、NONE 行也落库
- **`confirmOperation`**：mock `SignalLogEntity`，按 operationType switch 跑场景：
    - `OPEN` 推进 `FundStatus` + 写 `FundTransaction(source=BUY, amount=已知, shares=null, status=PENDING_NAV)` + 初始化 `holdingPeriodPeakNav`
    - `ADD_TIER_N` 写 `tierNAddedAt` + 写买入 FundTransaction（同上）
    - `TRAILING_STOP` 写卖出 FundTransaction（`source=SELL, shares=已知, amount=null, status=PENDING_NAV`）
    - `OVERRIDE_ADD` 同 `ADD_TIER_N` 但 `amount` 来自请求（可与 `suggestedMeasure.value` 不同）
    - 用户跳过/不回应 → 不写交易、不推进状态
    - 买入请求缺 `actualAmount` / 卖出请求缺 `actualShares` → 校验失败
    - `signalLogId` 指向不存在的信号 → 抛 `EntityNotFoundException`
- **`NavConfirmJob`**：mock `FundNavHistoryEntity` 当日净值，跑场景：
    - 买入 `PENDING_NAV` 行 → `shares = amount / nav` 回填、`status=CONFIRMED`
    - 卖出 `PENDING_NAV` 行 → `amount = shares × nav` 回填、`status=CONFIRMED`
    - 当日无净值行的 `PENDING_NAV` 保留不变（等次日再跑）
    - 撤单：`PENDING_NAV → CANCELLED`；转换撤单时两条 `relatedTransaction` 一起 CANCELLED
- **回测引擎**：单测固定净值序列场景，验证策略收益计算、三条基准线对比（收益 + 回撤）、`passed` 判定（含边界：收益刚好相等、回撤刚好等于 all-in、收益跑赢三条但回撤超 all-in 等）。
- **FundPosition 聚合**：构造多笔 `FundTransactionEntity`（含 `PENDING_NAV` / `CONFIRMED` / `CANCELLED` 混合、份额增减、转换两条腿），验证持仓份额 = Σ shares × direction（仅 CONFIRMED）、在途订单 = Σ PENDING_NAV、CANCELLED 不计入、持仓金额随传入净值变化。
- **不跑**端到端 Web 测试（这一期）。
