## 1. 概念与状态模型（需先对齐）

框架里"基金状态"和现有 `FundStatus`（待校准/持仓中/已归档）不完全是一回事，需要拆成两条互相独立的生命周期：

### 1.1 基金生命周期（`FundStatus`）

> 描述"这只基金在我账上的存在状态"，跟参数无关。

| 状态                    | 含义                                     | 进入条件     |
|-----------------------|----------------------------------------|----------|
| `PENDING_HOLDING` 未建仓 | 已纳入观察池，但没有买入                           | 新建基金     |
| `HOLDING` 持仓中         | 已经买入并持有份额                              | 首次建仓买入确认 |
| `CLEARED` 已清仓         | 持仓份额为 0（移动止盈分档卖光 / 逻辑破坏止损全清 / 再平衡减到 0） | 持仓份额归零   |

合法跃迁：

- `PENDING_HOLDING → HOLDING`
- `HOLDING → CLEARED`
- `CLEARED → PENDING_HOLDING`（清仓后又重新观察，允许，必须人工确认）
- `CLEARED` 是持仓状态机的自然终点，可以一直停在 `CLEARED`

> 注意：现有 `FundStatus` 里的 `PENDING_CALIBRATION` 是"参数没校准"的语义，不属于这条线，应迁移到下面的策略参数状态。
> `ARCHIVED` 不再作为 `FundStatus` 的值——归档通过软删除表达，见 1.1A。

### 1.1A 归档 = 软删除（独立于持仓状态机）

归档的真实场景是"我不想在这个平台看见这只基金了"（上错了、不想再看了、没买过也不打算买）。这不是持仓状态的一部分，而是**记录可见性
**，直接用 `AbstractEntity` 现有的 `@SoftDelete` + `deletedDate` 机制表达。

- **归档动作** = 执行软删除（`deletedDate = now`）
- **取消归档** = 恢复（`deletedDate = null`），状态回到归档前的样子
- 默认查询天然过滤已软删记录（Hibernate `@SoftDelete` 自动加 `WHERE deleted_date IS NULL`）
- 不引入 `visibility` / `archived` 这种冗余字段
- 软删 ≠ 物理删除：数据物理保留，只是从视图隐藏

**关键边界：**

1. **任意 `FundStatus` 都允许软删**，包括 `HOLDING`。不卡校验、不强制填原因——用户可以强制归档仍有持仓的基金（比如换平台管了，只是懒得删记录）。
2. **关联数据一起软删**：归档基金时，`FundTransactionEntity`、`SignalLogEntity`、`FundStrategyEntity`、
   `FundNavHistoryEntity` 中该基金的所有关联记录同步软删。恢复归档时一并恢复。理由：归档是"我不想看见这只基金了"
   ，关联数据是这只基金的衍生信息，跟着基金一起隐藏最自然，不留半残数据。
3. **归档不是删除账目事实**：软删只是隐藏，物理数据保留。如果以后真要物理清理过期归档数据，另起一个独立的清理任务，不混进归档动作里。

### 1.2 策略参数状态（`StrategyParamStatus`）

> 描述"这只基金绑定的纪律参数本身的状态"，跟持仓无关。

| 状态               | 含义                            | 进入条件            |
|------------------|-------------------------------|-----------------|
| `PENDING_CALIBRATION` 待校准      | 已建参数草稿，档位/金额还在调，不可用于推导        | 新建策略参数记录        |
| `CALIBRATED` 已校准 | 参数已对照该基金历史最大回撤分布定好，但尚未启用；**进入时自动跑过去一年回测并落 `StrategyBacktestEntity`** | 人工确认校准完成（同时自动回测）        |
| `EFFECTIVE` 已生效     | 当前生效中的参数，信号推导只看 `EFFECTIVE` 的那一份 | **通过回测验证后**人工激活 |
> 策略参数状态只有 3 态，**没有 `RETIRED`**。被新版参数替换的旧 `EFFECTIVE` 直接回退到 `CALIBRATED`；主动停用同样回退到 `CALIBRATED`。任期记账由 `FundStrategyActivationEntity.deactivatedAt` 表达，主表状态保持精简。原表第 4 行 `RETIRED` 已删。

合法跃迁：

- `PENDING_CALIBRATION → CALIBRATED → EFFECTIVE`
  - `PENDING_CALIBRATION → CALIBRATED`：人工确认校准完成时**自动跑过去一年窗口的回测**并落 `StrategyBacktestEntity`，CALIBRATED 自带一份回测结果
- `CALIBRATED → EFFECTIVE` **前置条件**：存在一份 `passed = true` 的回测结果（收益跑赢沪深300 + 一次性 all-in +
  等额定投三条基准线，且回撤 ≤ all-in）。calibrate 自动跑的那份 `passed = true` 即可直接激活；无有效回测结果不允许激活。
- `EFFECTIVE → CALIBRATED`（当有新参数被激活时，旧的回退 CALIBRATED；主动停用同此）
- `EFFECTIVE → PENDING_CALIBRATION`（`CLEARED → PENDING_HOLDING` 时该基金所有参数强制回退——清仓是分水岭，旧校准和旧回测都不可信，必须重新走 calibrate → 自动回测）
- `PENDING_CALIBRATION → CALIBRATED` 之后可直接丢弃草稿（靠软删，不另设退役状态）

### 1.3 关键规则：未绑定参数 → 不生成信号

- 一只基金未绑定任何 `EFFECTIVE` 的策略参数时，**每日定时任务直接跳过这只基金**，不落 `SignalLogEntity` 行，不做任何计算，也不抛异常。
- 这意味着信号引擎核心函数 `evaluateSignal` 的第一步永远是：先查 `EFFECTIVE` 参数，没有就 short-circuit 返回 `NONE`（reason=`NO_EFFECTIVE_STRATEGY`）。定时任务的遍历范围本身就是"有 EFFECTIVE 参数的基金"，这层 short-circuit 是兜底。
- 建仓信号是个特例：未建仓 + 已有 `EFFECTIVE` 参数 → 可推导；未建仓 + 无参数 → 仍然不推导（要求先把参数定好再讨论是否建仓，否则首笔
  10% 算不出来）。

### 1.4 两条状态线的组合关系

| FundStatus   | StrategyParamStatus        | 能做什么                                    |
|--------------|----------------------------|-----------------------------------------|
| `PENDING_HOLDING` | 无 / `PENDING_CALIBRATION` / `CALIBRATED` | 只能编辑参数，不出任何信号                           |
| `PENDING_HOLDING` | `EFFECTIVE`                   | 可推导建仓信号                                 |
| `HOLDING`    | 无 / `PENDING_CALIBRATION` / `CALIBRATED` | 不出信号（要求先把参数搞好）                          |
| `HOLDING`    | `EFFECTIVE`                   | 可推导加仓 / 卖出信号                            |
| `CLEARED`     | 任意                         | 不出信号（人工确认"继续观察"回 `PENDING_HOLDING` 后重新走建仓流程） |

> 软删（归档）的基金不出信号、不出现在默认列表，跟 `FundStatus` 正交，不在这张表里体现。

> 待确认：`CLEARED` 之后再次建仓，是回 `PENDING_HOLDING` 还是直接 `HOLDING`？倾向于"清仓后基金状态自动回 `PENDING_HOLDING`，参数保留
`EFFECTIVE`，下一次买入重新跑建仓首笔 10%"，这样档位状态、持有期高点都能自然清零。

## 2. 实体设计要点

### 2.1 `FundEntity`（基金产品本身）

只描述"是哪只基金" + 跟策略无关的**基金属性、市场事实和持有事实**，不存仓位事实。

| 字段                                                      | 类型                         | 说明                                                                         |
|---------------------------------------------------------|----------------------------|----------------------------------------------------------------------------|
| 继承自 AbstractEntity                                      |                            | id / version / 审计 / 软删                                                     |
| fundCode / fundName                                     | String                     |                                                                            |
| investmentTarget / operationMode / investmentPhilosophy | enum                       | 已有                                                                         |
| fundCategory                                            | FundCategory enum          | 宽基 / 行业 / 主动 / 混合（**基金本身属性**，跟参数版本无关；由原 `FundStrategy` 迁入）                 |
| fundSubType                                             | FundSubType enum, nullable | 数据源维度的基金分类（ETF/INDEX/INDEX_ENHANCED/ACTIVE），决定行情数据源和逻辑止损判定路径（见 CONTEXT.md） |
| benchmarkIndexCode                                      | String, nullable           | 跟踪/基准指数代码（如 `000300.SH`），指数类基金自动识别填入，主动类基金留空                               |
| status                                                  | FundStatus                 | 默认 PENDING_HOLDING（不再用 PENDING_CALIBRATION）                                |
| plannedTotalAmount                                      | BigDecimal, nullable       | 计划总仓位（纪律意图）；建仓时（PENDING_HOLDING → HOLDING）必填，HOLDING/CLEARED 状态下非 null     |
| openedAt                                                | Instant, nullable          | 首次建仓时间（持有事实，跟策略版本无关）；`CLEARED → PENDING_HOLDING` 时清空                       |

**移除（2026-06-24 修订，ADR-0001）**：`peakNav` / `holdingPeriodPeakNav` 两个字段。它们是 `fund_nav_history` 的 max
派生值，落库有净值修正、补录、job 异常三种失真风险。改为实时派生查询：

- `getPeakNav(fundId)` = `SELECT max(accumulated_nav) FROM fund_nav_history WHERE fund_id = ? AND deleted_date IS NULL`
- `getHoldingPeriodPeakNav(fundId)` = 同上加 `AND nav_date >= fund.openedAt`

> 原 `amount` / `shares` 字段也移除：事实仓位由 `FundTransactionEntity` 聚合得出。

> `plannedTotalAmount` / `openedAt` 都从 `FundStrategyEntity` 搬过来——它们跟策略参数版本无关，是基金本身的属性或持有事实。换参数版本不影响这些值。

> **`FundStatus` 变更不走日志表**：变更不频繁，无审计需求。`openedAt` 是持有事实直接存在 `FundEntity` 上，不需要从日志表反推。

### 2.2 `FundStrategyEntity`（策略参数 + 该版本下的运行时状态）

只放跟**策略参数版本相关**的字段。基金属性和全局硬约束不在这里。

**身份与生命周期**

- `fundEntity` ManyToOne
- 不预留 `userId`（本期单用户场景，零迁移成本省掉）
- `status`（`StrategyParamStatus`）见 1.2
- ~~/`fundCategory`~~ —— **改挂 `FundEntity.fundCategory`**：基金分类是基金本身属性（用于查默认档位和全局硬约束上限），跟参数版本无关，换参数版本不应该重选类型。原 `FundStrategy.fundCategory` 字段已迁出。

**4 个可调参数**（`singlePositionLimit` 已升级为全局配置，不在这里）

- `tier1Drawdown` ~ `tier4Drawdown`（BigDecimal，回撤百分比，如 0.08 = 8%）
- `tier1Ratio` ~ `tier4Ratio`（BigDecimal，加仓比例，占 `plannedTotalAmount` 的百分比，如 0.15 = 15%；**四档之和 ≤ 90%**）
- `weeklyCoolDownThreshold`（BigDecimal，单周跌幅冷静阈值，如 0.10 = 10%；见下方说明）
- `stopLossPullbackPercent`（BigDecimal，移动止盈回落幅度，如 0.08 = 8%）

**运行时状态**

- `tier1AddedAt` ~ `tier4AddedAt`（Instant, nullable，方案B 四档**实际执行**时间戳，null = 未加；**只在用户通过 POST
  /operations 确认执行时写入**，信号生成只读不写）

**不在主表存的字段**

- `openedAt` → 搬到 `FundEntity`（持有事实，跟策略版本无关）
- `activatedAt` / `deactivatedAt` → 不存主表，走 `FundStrategyActivationEntity` 激活表（见《推荐架构方案》2.7）

> 一只基金允许同时存在多份策略参数（PENDING_CALIBRATION/CALIBRATED/EFFECTIVE 共存），但任一时刻同一只基金最多 **1 份 EFFECTIVE**。靠"
> 激活时把旧的回退到 CALIBRATED"的服务逻辑保证，DB 层加部分唯一索引兜底。

**`weeklyCoolDownThreshold` 语义说明**：

- 用于检测"跌得太快"的异常情况，触发**强制冷静强提示**
- **算法（2026-06-24 修订）**：取最近 5 个交易日的累计净值（含今日，但 14:50 信号生成时今日净值未公布，实际是 [T-5, T-1]
  区间），算两点跌幅 `(5天前累计净值 - 最近累计净值) / 5天前累计净值`；超过 `weeklyCoolDownThreshold` → 在加仓信号的
  `warnings` 加 `WEEKLY_COOLDOWN`："单周跌幅 X% 超过冷静阈值 Y%，建议暂停加仓等周线企稳"。**用累计净值不用单位净值**
  （避免分红除权虚高）。
- 这是**强提示，不是硬约束**——人工可 override，但要记录原因
- 跟"破位观望"并存：强制冷静看跌幅速度，破位观望看趋势+量，两者是不同维度
- 数据不足降级：新建仓基金净值历史不足 5 个交易日的，加 `INSUFFICIENT_DATA_FOR_COOLDOWN` warning，不阻断加仓
- 主动基金的逻辑止损判定也复用同一算法（见 CONTEXT.md）
- 默认值参考（按 `FundCategory`）：宽基 8% / 行业 12% / 主动 10% / 混合 10%

**`tier1~4AddedAt` 写入时机**（重要）：

- 信号生成（`generateDailySignals`）只读不写 `tierNAddedAt`（例外：反弹清空是状态机自动行为，随信号生成一起写）
- `POST /operations` 确认执行时才写：对应 `SignalLog.signalType=ADD, triggerTier=N` → 写 `tierNAddedAt = now`
- 用户不回应 → `tierNAddedAt` 仍是 null → 下次还在该档会再提示
- 信号生成里唯一允许的写：**反弹清空**（当前档比已加最深档浅 → 清空更深档的 `tierNAddedAt`），这是状态机自动行为，不依赖用户操作

**反弹清空缓冲带（2026-06-24 修订，ADR-0003）**：清空判定加 0.5% 缓冲——`drawdown < tierNDrawdown - 0.005` 才认为该档被"真正"
脱离并清空（及更深的档）。加档侧不变（`drawdown >= tierNDrawdown` 即触发）。目的是防止净值在档位边界震荡导致"加 - 清 - 加"
频繁抖动。清空动作在 SignalLog.warnings 里加 `TIER_CLEARED` 提示，不单独产生 SignalLog 行。常量 `TIER_CLEAR_BUFFER = 0.005`
落 `support/` 层。

### 2.3 `FundTransactionEntity`（基金交易 · 改名自 `FundFlowEntity`）

原 `FundFlowEntity` 改名为 `FundTransactionEntity`，承载**基金交易事件**——买入、卖出、转入、转出。只合并了原 `StrategyOperationEntity` 的 `signalLog` 字段（手动操作和转账该字段为 null）；**不复用 `operationType`**——策略动作（建仓/加档几/移动止盈/逻辑止损/再平衡）由 `signalLog` 关联回读 `SignalLog.signalType + triggerTier` 得到，不在交易表上冗余。

**字段**

- `fund` ManyToOne
- `source`（`FundTransactionSource`：INCREASE / DECREASE / TRANSFER_IN / TRANSFER_OUT）
- `status`（`FundTransactionStatus`：PENDING / CONFIRMED / CANCELLED）
- `shares`（BigDecimal, nullable）—— 买入下单时 null（等净值回填）；卖出下单时已知；转账已知；回填后永不改
- `amount`（BigDecimal, nullable）—— 买入下单时已知；卖出下单时 null（等净值回填）；转账已知；回填后永不改
- `nav`（BigDecimal, nullable）—— 成交净值，净值确认后回填；下单时 null
- `confirmTime`（Instant, nullable）—— 净值确认时间
- `signalLog`（ManyToOne, nullable）—— 由信号触发的交易才填；手动 / 转账 null
- `relatedTransaction`（OneToOne, nullable）—— 基金转换时指向另一条腿（A 转 OUT + B 转 IN 互指）；其他场景 null

**交易时序与状态机**

```
PENDING  ── NavConfirmJob 跑完 ──→  CONFIRMED    (买入：回填 shares=amount/nav；卖出：回填 amount=shares×nav)
PENDING  ── 用户撤单 ──→           CANCELLED
```

- 下单那一刻写入已知侧：买入 `amount` 已知、`shares=null`；卖出 `shares` 已知、`amount=null`；`nav=null`、`status=PENDING`
- 当晚 `NavConfirmJob` 跑：查 `FundNavHistoryEntity` 拿当日净值，回填另一侧 + `nav` + `confirmTime`，转 `CONFIRMED`
- 回填后字段不再改变

**持仓计算**

```
持仓份额 = Σ( shares × direction ) WHERE status = 'CONFIRMED'     # INCREASE/TRANSFER_IN=+1, DECREASE/TRANSFER_OUT=-1
在途订单 = Σ( ... ) WHERE status = 'PENDING'                  # 前端单独展示
持仓金额 = 持仓份额 × 当前净值                                       # 实时算，不存
```

> "今天净值上涨导致钱变多" 不需要任何交易记录，`持仓金额 = 持仓份额 × 当前净值` 实时算即可。只有真正发生买卖时才写交易。
>
> `FundTransactionEntity` 合并了原 `StrategyOperationEntity` 的职责。`OVERRIDE_SKIP` 不写交易（与被动忽略在数据上不区分）；手动 / 转账的策略上下文字段为 null。
>
> 转换（基金 A → 基金 B）= 两条交易（TRANSFER_OUT + TRANSFER_IN）通过 `relatedTransaction` 互指，撤单时一起 CANCELLED。

### 2.4 新增 `SignalLogEntity`（每日信号快照）

每日 14:50 定时任务对每只绑定 `EFFECTIVE` 策略的基金跑一遍信号引擎，把当天的信号结果冻进一张快照表。信号是 14:50 那一刻的市场切片，指标/净值收盘后无法复算，必须当时落库。

字段：

- `fund` ManyToOne
- `userFundStrategy` ManyToOne（当天生效的参数版本）
- `signalDate`（Instant，14:50 那个交易日）
- `signalType`（`SignalType`：NONE / BUILD / ADD / SELL）
- `triggerNav`（触发时净值，signalType ≠ NONE 时）
- `triggerTier`（触发档位 1~4，ADD / 移动止盈 SELL 时有效）
- `coefficient`（最终系数，ADD 时）
- `suggestedMeasure`（`Measure` 值对象 `{value, measureUnit}`：BUILD/ADD 时 `measureUnit=AMOUNT`、`value`
  是建议金额，买入按金额下单；SELL 时 `measureUnit=SHARES`、`value` 是建议份额，卖出按份额下单；NONE 时为 null）
- `reason`（NO_EFFECTIVE_STRATEGY / TIER_HIT / COOLDOWN / WATCH_AND_WAIT / TRAILING_STOP / LOGIC_BROKEN / REBALANCE /
  MIN_HOLD_DAYS_NOT_MET / NO_TIER_TO_SELL / INSUFFICIENT_MARKET_DATA / ...）
- `warnings`（JSONB，强提示列表）
- `hardConstraintBreaches`（JSONB，命中硬约束被截断的说明）
- `generatedAt`（Instant，任务执行时间）

**唯一性约束**：`(fund_id, signal_date)` 在未软删记录里唯一——每只基金每个交易日最多一行（含 NONE）。每日一行/基金，量级和 `fund_nav_history` 同级，兼作"系统每天评估过这只基金"的审计轨迹。

> **被动忽略可追踪**：`signalType ≠ NONE` 且无对应 `FundTransactionEntity` 的 SignalLog 行 = 系统提示了但用户没回应（含主动 SKIP 和被动忽略，两者在数据上不区分）。`LEFT JOIN fund_transaction ON signal_log.id = fund_transaction.signal_log_id WHERE transaction.id IS NULL AND signal_type <> 'NONE'` 直接列出。
>
> **`suggestedMeasure` 用 `Measure` 值对象**：`{value, measureUnit}` 带 `MeasureUnit`（AMOUNT/SHARES）的值对象，由
`signalType` 决定 measureUnit——BUILD/ADD 时 measureUnit=AMOUNT（建议金额，买入按金额下单），SELL 时
> measureUnit=SHARES（建议份额，卖出按份额下单），NONE 时整个对象为 null。值对象定义和 `MeasureUnit` 枚举说明见《推荐架构方案》2.4
> 末尾和 3.9。

### 2.5 新增 `StrategyBacktestEntity`（回测结果）

`CALIBRATED → EFFECTIVE` 的前置条件：必须有一份回测结果同时跑赢三条基准线。

字段：

- `userFundStrategy` ManyToOne（被回测的参数版本）
- `backtestStart` / `backtestEnd`（回测窗口）
- `strategyReturn` / `strategyMaxDrawdown`（策略表现）
- `benchmarkHs300Return` / `benchmarkAllInReturn` / `benchmarkDcaReturn`（三条基准线收益率）
- `benchmarkHs300MaxDrawdown` / `benchmarkAllInMaxDrawdown` / `benchmarkDcaMaxDrawdown`（三条基准线最大回撤）
- `passed`（boolean，是否通过：收益跑赢三条且策略回撤 ≤ all-in 回撤）

> **通过判定（`passed`）**：策略最终收益率同时大于三条基准线收益率（沪深300同期、一次性 all-in、等额定投），**且**
> 策略最大回撤 ≤ all-in 同期最大回撤。两条都满足才算通过，落 `passed = true`。
>
> 收益率条件是策略的收益价值证明（直接买基准就行，策略必须更强）；回撤条件是纪律策略的设计底线（分档加仓的回撤不该比一次性
> all-in 还大，否则逻辑破产）。回撤用 `≤` 而非 `<`，避免临界值抖动导致反复激活失败。不要求回撤跑赢三条基准——定投/沪深300
> 回撤可能很小，门槛过高会让回测前置形同虚设。
>
> **不存 `initialCapital`**：策略参数是比例制（`tier1~4Ratio` 占 `plannedTotalAmount`
> 的百分比），回测在比例空间里算——策略收益率 = Σ ratio_i × (期末净值/加仓时净值_i − 1)，`plannedTotalAmount`
> 被约掉，与资金盘子绝对值无关。三条基准线同理。所以回测不需要初始资金入参，也不需要从 `fund.plannedTotalAmount` 取值。
>
> **不存 `backtestDataSnapshot` / `executedAt`**：净值序列是回测输入，接口接受入参，复跑重新传即可，不冻档（结果字段已冻，足够审计）；执行时间由
`AbstractEntity.createdDate` 覆盖，不重复记录。
>
> **回测只验单基金维度的硬约束**（单品种仓位上限、单次加仓 ≤ 剩余可用 50%、7
> 天内不赎回）。跨基金维度的单类型仓位上限和总仓位上限不在回测里验——单基金回测没有全局资金盘子作分母——这两条留到运行时（
`SignalGenerationJob`）校验。

## 3. 信号推导服务边界

### 3.1 `DisciplineStrategyService` 的入口契约

两类入口分开：**生成**（定时任务调）和**确认**（用户调）。信号引擎核心 `evaluateSignal` 是无状态纯函数，两类入口都复用它。

`evaluateSignal(fund, strategy, marketIndicators, capitalContext) → SignalResult` 的固定步骤：

1. 查 `FundEntity` → 校验 `FundStatus` 是否允许出该类信号（见 1.4 表格）；不允许 → `signalType=NONE`。
2. 校验 `EFFECTIVE` 策略存在 → 没有就返回 `signalType=NONE`，`reason="NO_EFFECTIVE_STRATEGY"`。
3. 有了再进入框架算法（档位 / 系数 / 硬约束 / 强提示）。

`generateDailySignals(signalDate)`：遍历所有"绑定 EFFECTIVE 策略 + 未软删"的基金，对每只调 `evaluateSignal`，结果落一行 `SignalLogEntity`（含 NONE）。

`confirmOperation(signalLogId, confirmRequest)`：写 `FundTransactionEntity`（买入写 amount / 卖出写 shares，另一侧 null，`status=PENDING`）+ 推进档位 / `FundStatus`。不复算信号——建议侧字段已在 `SignalLogEntity` 里。`OVERRIDE_SKIP` 不调本接口、不写交易（与被动忽略在数据上不区分）。

### 3.2 市场指标输入

年线状态、周MACD、成交量、单周跌幅、60日新高这些**本期接入东方财富/天天基金公开接口**（ADR-0002，原"半自动灌入"定位已废弃）。由定时任务在
14:50 前通过 `MarketIndicatorProvider` 拉取，落 `market_indicator_snapshot` 表当缓存。后端只负责：

- 套档位、查表算系数、合成 + clamp、跑硬约束、出建议金额、出"是否破位观望/强制冷静"等强提示。

**三条数据线**：

- 基金净值历史（`pingzhongdata.js` 的 `Data_netWorthTrend` / `Data_ACWorthTrend`）
- 基金字典（`fundcode_search.js` 全量约 2 万条，用于 `fundSubType` 和 `benchmarkIndexCode` 自动识别）
- 指数 K 线（`push2his.eastmoney.com`，用于量能类指标）

**基金分类自动识别**：只做名称启发式（方法 A，名称含"沪深300/中证500/创业板"等关键词映射到对应指数代码）+ 兜底 ACTIVE（方法
C）；**跳过持仓股票重合度反推**（方法 B，复杂度高，留给将来）。

**限流与降级**：东方财富对 IP 有限速（约每秒 2-3 次），用 `Semaphore` 或 `Bucket4j` 节流，加
`Referer: https://fund.eastmoney.com/` 头避免反爬。指标缺失时不出信号（
`signalType=NONE, reason=INSUFFICIENT_MARKET_DATA`）。

> `MarketIndicatorProvider` 接口保留，按 `fundSubType` 分派不同实现：ETF/指数走指数 K 线，主动走净值跌幅。

### 3.3 端点（先用最小集）

- `PUT /api/funds/{fundId}/strategies` — 新建/更新一份 `PENDING_CALIBRATION` 参数
- `POST /api/funds/{fundId}/strategies/{id}/calibrate` — `PENDING_CALIBRATION → CALIBRATED`（同时自动跑过去一年回测，落 `StrategyBacktestEntity`）
- `POST /api/funds/{fundId}/strategies/{id}/activate` — `CALIBRATED → EFFECTIVE`（同时把旧 EFFECTIVE 回退 CALIBRATED，并回填 `FundStrategyActivationEntity.deactivatedAt`）
- `GET /api/funds/{fundId}/strategies/active` — 查当前生效参数和运行时状态
- `GET /api/funds/{fundId}/signals/today` — 查当天 `SignalLogEntity`（已由 14:50 定时任务生成）
- `GET /api/funds/{fundId}/signals?from=&to=` — 历史信号区间查
- `GET /api/signals/pending` — 跨基金查所有未回应的信号（前端工作台用）
- `POST /api/admin/signals/generate` — 手动触发当日信号生成（管理/调试用，正常由定时任务触发）
- `POST /api/funds/{fundId}/operations` — 人工回应某条信号，写 `FundTransactionEntity`
  ，推进档位 / `FundStatus`。请求体里带 `signalLogId` 指向被回应的那条 `SignalLogEntity`；买入带 `actualAmount`、卖出带 `actualShares`
- `POST /api/transactions/{id}/cancel` — 撤单：`PENDING → CANCELLED`；基金转换撤单时两条 `relatedTransaction` 一起撤
- `POST /api/admin/transactions/confirm-nav` — 手动触发 `NavConfirmJob`（管理/调试用，正常由每晚定时任务触发）

## 4. 硬约束的归属

框架第九节的 6 条硬约束，落到代码里要拆清归属：

| 约束                  | 归属层  | 说明                                                                         |
|---------------------|------|----------------------------------------------------------------------------|
| 单品种仓位上限 20%（行业 15%） | 信号服务 | 加仓建议金额计算时直接截断；**存量超限触发再平衡 SELL 信号**                                        |
| 单类型仓位上限 30%         | 信号服务 | 需要"同类型总仓位"聚合查询                                                             |
| 总仓位上限 80%           | 信号服务 | 需要"可投资金"作为输入参数；分母取 `UserConfig.totalInvestableCapital`                     |
| 单次加仓 ≤ 剩余可用 50%     | 信号服务 | 算完建议金额后再截断                                                                 |
| 7 天内不赎回             | 卖出信号 | 窗口 **5 个交易日**（非自然日 7 天）；起算 = `max(openedAt, tier1~4AddedAt)`；逻辑止损豁免；手动卖出不卡 |
| 不借钱加仓               | 不入代码 | 这条是纪律，不在系统里建模                                                              |

**SELL 信号的三种类型与优先级**（2026-06-24 修订）：

| 类型     | reason          | 触发条件                                                                                                  | 卖出份额                                    | 7 天豁免 |
|--------|-----------------|-------------------------------------------------------------------------------------------------------|-----------------------------------------|-------|
| 逻辑破坏止损 | `LOGIC_BROKEN`  | ETF/指数：净值跌破年线 + 周MACD绿柱扩大 + 跟踪指数放量下跌；主动：净值跌破年线 + 周MACD绿柱扩大 + 单周跌幅 > 阈值                                | 当前持仓全部份额                                | **是** |
| 移动止盈   | `TRAILING_STOP` | 从 `holdingPeriodPeakNav` 回落 N × `stopLossPullbackPercent`（N=1~4）；从深档往浅档查找首个 `tierNAddedAt` 不为 null 的档 | 该档对应加仓交易的实际入账份额（A1 规则）；触发 4 档时连同建仓份额一起卖 | 否     |
| 再平衡    | `REBALANCE`     | 单只占比 > `singlePositionLimit`                                                                          | `(当前占比 - 上限) × 总权益持仓金额 / 最近净值`          | 否     |

**优先级**：逻辑止损 > 移动止盈 > 再平衡。`evaluateSignal` 按此顺序检查，命中即返回，一只基金每日一行 SignalLog。

**新增表**：`user_config`（本期单用户场景一行，存 `totalInvestableCapital`）、`trading_calendar`（A 股交易日历，5 交易日窗口判定用）、
`market_indicator_snapshot`（行情数据落库缓存）。

**新增常量（落 `support/HardConstraintConfig`）**：

- `BUILD_RATIO = 0.10`（建仓首笔比例）
- `TIER_CLEAR_BUFFER = 0.005`（反弹清空缓冲带，ADR-0003）
- `LOGIC_BROKEN_EXTRA_DRAWDOWN`（已废弃；逻辑止损改走三指标判定，不再用极端回撤兜底）

## 6. 待拍板的开放问题

### 已确认

- ✅ **同一只基金允许多份历史参数共存**（原问题 3）：保留版本历史，便于追溯参数演进。
- ✅ **不预留 `userId`**（原问题 4，修订）：本期单用户场景零迁移成本省掉；将来引入多用户时再加 nullable 列即可。
- ✅ **默认档位表放代码常量**（原问题 5）：等参数化需求出现再迁。
- ✅ **`FundFlowEntity` 改名为 `FundTransactionEntity`，维护份额不是金额**（新增/修订）：改名反映"基金交易事件"语义；金额会随净值每日变动，份额只在买卖时才变。`持仓金额 = 持仓份额 × 当前净值`
  实时算。
- ✅ **交易时序：下单写已知侧，净值确认后回填另一侧**（新增）：基金交易现实是 14:50 信号 → 15:00 前下单 → 当晚净值公布。买入下单时知道 `amount`、不知道 `shares`（净值未公布算不出）；卖出下单时知道 `shares`、不知道 `amount`。`FundTransactionEntity` 同行承载：下单写入已知侧 + `status=PENDING`，当晚 `NavConfirmJob` 跑完回填另一侧 + `nav` + `confirmTime`，转 `CONFIRMED`。"事实账目回填后永不改变"原则保留（下单时 null → 净值确认后填一次 → 不再改）。
- ✅ **`StrategyOperationEntity` 合并进 `FundTransactionEntity`**（新增）：原操作表的策略上下文只保留 `signalLog` 字段——由信号触发的交易才填，手动 / 转账为 null。**不复用 `operationType`**——策略动作（建仓/加档几/移动止盈/逻辑止损/再平衡）由 `signalLog.signalType + triggerTier` 关联回读表达，不在交易表上冗余。`OVERRIDE_SKIP` 不写交易（与被动忽略在数据上不区分）。原 1:0..1 关系简化为：`SignalLog : FundTransaction = 1:0..1`（一条信号最多触发一条交易，未触发即被动忽略或主动跳过，两者不区分）。
- ✅ **`FundTransactionSource` 沿用 + `FundTransactionStatus` 重做**（新增）：`FundFlowSource` 改名为 `FundTransactionSource`，**沿用现有五值不动**（`INCREASE / DECREASE / TRANSFER_IN / TRANSFER_OUT / INVEST`）——`INCREASE`/`DECREASE` 承担买卖，语义在系统中已稳定，不强行改名为 BUY/SELL 以免做无意义改动；`INVEST` 保留枚举值不用，将来做定投时每期落一条 `INVEST`（不做定投的话另起 `DcaPlanEntity` 表）；转换用 `TRANSFER_IN` + `TRANSFER_OUT` 两条交易通过 `relatedTransaction` 互指表达。`FundFlowStatus` 改名重做值为 `PENDING / CONFIRMED / CANCELLED`。
- ✅ **不区分主动 SKIP 和被动忽略**（新增）：用户主动点跳过和不理会信号在数据上等价——都是"信号有行、FundTransaction 无对应"。前端"待回应列表"用 `signal_type <> 'NONE' AND signal_log.id NOT IN (SELECT signal_log_id FROM fund_transaction WHERE signal_log_id IS NOT NULL)` 查。要区分的话以后在 `SignalLogEntity` 加 `responseType` 字段，本期不做。
- ✅ **`overrideReason` 字段删除 + override 不留痕**（新增）：用户改了建议金额下单时，`FundTransaction.amount` 就是实际下单值，系统不留痕——不记录是否 override、不做与 `SignalLog.suggestedMeasure.value` 的差值记录。`OVERRIDE_SKIP` 的 reason 一并删除（跳过即无记录）。
- ✅ **归档时关联数据一起软删**（新增）：交易、信号快照、参数版本、净值历史同步软删；恢复归档时一并恢复。
- ✅ **激活参数前必须先回测且跑赢三条基准线**（新增）：沪深300、一次性 all-in、等额定投，三条同时跑赢才允许
  `CALIBRATED → EFFECTIVE`。**calibrate 时自动跑回测**并落 `StrategyBacktestEntity`（窗口固定"过去一年"，不足一年时降级到"自基金成立至今"——`backtestStart` 字段实际写入最早可用日期，不引入新字段；`passed` 判定跟窗口长度脱钩），CALIBRATED 自带一份回测结果——`passed = true` 可直接激活，`passed = false` 提示调参或换窗口重测；人工也可调 `backtest` 接口用自定义窗口复跑。
- ✅ **`CLEARED` 不是终态**（新增）：`CLEARED` 是持仓状态机的自然终点，可以一直停着；想再买回来走 `CLEARED → PENDING_HOLDING`
  （人工确认），重新跑建仓流程。
- ✅ **归档 = 软删除**（新增）：归档不是持仓状态的一部分，直接用 `@SoftDelete` 表达。`FundStatus` 去掉 `ARCHIVED` 值。
- ✅ **任意 `FundStatus` 都允许软删**（新增）：包括 `HOLDING`，不卡校验、不强制填原因。
- ✅ **归档时关联数据一起软删**（新增）：流水、信号快照、操作记录、参数版本、净值历史同步软删；恢复归档时一并恢复。
- ✅ **`FundEntity.status` 复用为 `FundStatus`**（原问题 1）：拿掉 `PENDING_CALIBRATION`，改用 `PENDING_HOLDING` / `HOLDING` /
  `CLEARED` 三个值。改动成本接近 0。
- ✅ **`singlePositionLimit` 升级为全局配置**（新增）：是全平台硬约束规则，不是每基金策略意图。落代码常量

    - 四类基金共用同一组比例，差异化体现在回撤阈值上（已在 `DefaultTierTable`），比例不需要再分类型调
