# FundPilot · 基金定投移动止盈策略上下文

本上下文描述 FundPilot 后端"基金定投移动止盈策略"领域的核心术语与边界。它是领域语言的词典，不是规格说明书。实现细节、字段定义、状态机见
`backend/docs/` 下的设计文档。

## 信号与交易动作

**SignalType（信号类型）**:
系统每日 14:50 对绑定 `EFFECTIVE` 策略的基金评估后冻进 `SignalLogEntity` 的策略意图，站在**用户视角的策略动作**。两值：
`NONE`（无建议）/ `SELL`（止盈卖出）。定投买入不经信号引擎（定时任务自动扣款），信号引擎只管止盈。
_Avoid_: BUILD / ADD（择时建仓加仓信号，已废弃）；DECREASE（DECREASE 是账目方向，不是策略意图）

**FundTransactionSource（交易来源）**:
账目层份额变化方向的中性描述，五值：`INCREASE` / `DECREASE` / `TRANSFER_IN` / `TRANSFER_OUT` / `INVEST`。与 `SignalType` 是
**两个不同维度**——信号描述"系统建议什么策略动作"，来源描述"账目份额怎么动"。其中 `INVEST` 是定投买入，由系统定时任务自动产生；
`INCREASE` 是手动加仓。一只基金的交易来源沿用现有五值不收敛。
_Avoid_: BUY / SELL（语义已被 SignalType 占用，避免歧义）

## 定投纪律

**定投（DCA）**:
等额定投，每期金额用户输入（基金级字段），无上限、无计划期数。定时任务自动扣款产生 `INVEST` 交易，**不经信号引擎**。定投是纯机械纪律，
不择时、不预测，到点扣款。
_Avoid_: 智能定投（估值加权，又把择时请回来）；趋势定投（看均线决定是否扣款，本质择时）

**定投暂停与恢复（DCA Pause/Resume）**:
按 `fundCategory` 分派：行业基金止盈后若市场继续跌、收益率重新跌回 10% 以下才恢复定投——止盈后市场转跌时暂停定投，避免下跌途中
继续加仓，收益跌回 10% 以下（市场冷却）才恢复。宽基基金**一直定投不停**，即使止盈也继续买——宽基永远可以继续买，不像行业有周期边界。
_Avoid_: 行业止盈后继续定投（下跌途中加仓，违背"止盈落袋"精神）；宽基止盈后暂停（宽基长牛，暂停会错过低位积累）

**扣款周期（DCA Schedule）**:
每月最后一交易日扣款，等额，金额 = 用户输入。与 dca 基准算法的扣款节奏对齐，保证回测基准与实盘同口径。
_Avoid_: 每月固定日（与 dca 基准节奏不一致，回测失真）

**滚动加权平均成本（Rolling Weighted Average Cost）**:
收益率计算的分母，随每次买入（定投 INVEST / 手动 INCREASE / 转入 TRANSFER_IN）加权更新——
`新成本 = (旧成本×旧份额 + 本次amount) / 新旧份额之和`。卖出不改成本。复用 `FundEntity.costPerShare` 语义。是收益率与市值计算的核心输入。
_Avoid_: 持仓成本总额（旧定义已废弃）；穿透交易表实时派生（成本是持仓属性应存储）

## 移动止盈

**历史最高市值 High（Peak Market Value）**:
移动止盈的基准。每日更新：`High = max(昨日 High, 今日市值)`。**High 永远不会下降**——市值创新高时上移，市值回落时保持。
是"让利润奔跑"的锚点：止盈线挂在 High 上，只有真正新高才抬升止盈线，回调不降。
_Avoid_: 用收益率峰值（旧方案已推翻，市值峰值更直观且与止盈线计算同源）；在 FundEntity 上存字段（派生值落库会失真，实时算 max(市值序列)）

**当前收益率（Current Yield）**:
公式 = `(当前市值 - 累计投入) / 累计投入`。累计投入 = 历次买入金额之和。用于判定启动门槛与回撤比例分级。随市值每日变动。
_Avoid_: 用净值涨跌代替收益率（净值涨跌不含成本信息，不能反映真实盈亏）

**启动门槛（Activation Threshold）**:
收益率达启动门槛才启动移动止盈，开始记录 High 与止盈线。按 `fundCategory` 分派：行业基金 40%，宽基基金 50%。收益率不到门槛
什么都不做，继续定投。门槛过滤掉小赚即止盈的噪音——宽基 30% 收益很常见（牛市常涨 80%、100%），太早启动容易卖飞，故宽基门槛更高；
行业波动更大，40% 才算趋势确立。
_Avoid_: 启动门槛太低（小赚即止盈，错过后续大涨）；不要启动门槛（任何盈利都追踪，频繁误触）；宽基与行业共用门槛（宽基 30% 太常见，会过早止盈）

**止盈线（Take-Profit Line）**:
公式 = `High × (1 - 回撤比例)`。**止盈线只会上移不会下降**——High 上移时止盈线同步上移，High 不变时止盈线钉住。跌破止盈线才触发卖出。
_Avoid_: 止盈线随市值下降（违背"让利润奔跑"，会过早卖出）

**回撤比例分级（Tiered Pullback Ratio）**:
回撤比例不固定，按历史最高收益率分级——收益越高，允许回撤越大，避免高收益时被正常波动洗出。按 `fundCategory` 分派两套：

行业基金（波动大，回撤档较窄，代表值 12%）：

| 历史最高收益率 | 回撤比例 |
|---|---|
| 40~60% | 10% |
| 60~90% | 12% |
| 90~130% | 14% |
| 130% 以上 | 17% |

宽基基金（如沪深300、中证A500，一年 10~20% 回调很正常，回撤档更宽，代表值 18%，避免每年都卖一次）：

| 历史最高收益率 | 回撤比例 |
|---|---|
| 50~80% | 15% |
| 80~150% | 18% |
| 150% 以上 | 20% |

理由：赚 100% 时一天跌 8% 很正常，止盈太近容易卖飞；宽基回调比行业更频繁更浅，回撤档必须更宽，否则基本每年都触发。分级表第一档起点
对齐启动门槛（行业 40%、宽基 50%）。
_Avoid_: 固定回撤比例（高收益时被正常波动洗出，低收益时回撤过大错过止盈）；宽基用行业的回撤档（宽基回撤档窄每年都触发，频繁卖飞）

**止盈触发与卖出（Take-Profit Trigger）**:
收盘价连续 2 个交易日低于止盈线才执行卖出（避免一天假跌破）。每次卖出比例按 `fundCategory` 分派：行业基金每次卖当前持仓的 **25%**，
宽基基金每次卖 **20%**（宽基更值得长期拿，每次少卖）。不全卖，分批落袋。卖出后以剩余仓位重新记录 High，下一轮移动止盈重新开始。
_Avoid_: 一次性全赎（放弃分批落袋与后续上涨）；单日跌破即卖（假跌破频繁交易）；宽基用 25%（宽基该长拿，每次少卖）

**最大减仓上限与底仓保留（Max Reduction Cap & Floor Position）**:
按 `fundCategory` 分派：行业基金**最多累计卖出 75%，至少保留 25% 底仓**；宽基基金**最多累计卖出 60%，至少保留 40% 底仓**。
两类基金都留底仓——宽基最终会上涨留 40% 吃长牛，行业留 25% 防周期误判（行业也可能有第二波）。累计卖出比例达上限后，
即使再次跌破止盈线也不再卖，只等新高重置。底仓保留是硬约束，不随止盈轮次清零。
_Avoid_: 行业可清仓至0（行业也可能有第二波，清仓会错过）；宽基也清仓（违背"宽基长拿"精神，错过长期上涨）；底仓比例太高（失去止盈落袋意义）

**卖出冷却期（Post-Sell Cooldown）**:
每次卖出后至少等待 20 个交易日，再判断下一次止盈。防止连续卖出——卖出后市场可能继续震荡，冷却期避免在震荡中把仓位快速清光。
_Avoid_: 无冷却期（连续卖出，几轮就把仓位清光，失去分批意义）

**止盈后重置（Post-Take-Profit Reset）**:
每次卖出后，以剩余仓位重新记录 High（旧 High 作废，从卖出后剩余市值起算新高）。成本基础、收益率随之重算（份额减少了）。
新一轮移动止盈从新 High 开始，重新等启动门槛与止盈线触发。
_Avoid_: 沿用旧 High（仓位已变，旧 High 与新仓位不匹配，止盈线失真）

**极端行情保护（Extreme Market Protection）**:
移动止盈适合正常回调，行业基金景气反转常是快速回撤。两条加速止盈规则，独立于移动止盈线：
① 单日下跌 ≥ 7%（接近跌停幅度）且已盈利 → 提前卖出 10~20% 仓位，降低尾部风险；
② 连续 3 个交易日累计跌幅 ≥ 12% → 即使未触及移动止盈线，也执行一次额外减仓。
理由：半导体/AI/新能源等主题景气度反转往往快速回撤，不等止盈线就减仓能少吐利润。
_Avoid_: 只靠移动止盈线（快速回撤时等跌破已晚，吐回过多利润）

**SELL 信号优先级**:
一只基金每日一行 SignalLog，SELL 信号最多一类。`evaluateSignal` 按顺序检查：极端行情保护 > 移动止盈。命中即返回。
`reason` 两值：`EXTREME_MARKET_PROTECT`（极端行情保护）/ `TRAILING_STOP`（移动止盈触发）。
_Avoid_: 同日多类型 SELL 信号叠加（违反"一只基金每日一行"的唯一性约束）

## 策略寻优

**寻优（Optimize）**:
与「校准（Calibrate）」正交的动作——校准是"验证给定参数是否达标"，寻优是"从默认基准出发自动生成候选参数"。寻优产物（最优参数草稿）再走一次标准 calibrate 验证落库，形成"寻优→校准"链条，不新增状态机分支。
_Avoid_: 自动校准（与现有 calibrate 动作重叠，混淆状态机）

**风险调整收益（Risk-Adjusted Return）**:
口径 = 策略收益 / 策略最大回撤（类 Calmar），既是寻优 train 集的择优标尺，也是 test 集 passed 判定的回撤维度——策略 Calmar 须 ≥ dca Calmar。
train 侧在候选间排序选最高，test 侧与 dca 比较定通过；同源口径，择优标尺与验证标尺闭环。
_Avoid_: 纯收益率最大化（可能选出回撤极大的激进参数）

**dca 基准对照口径**:
策略本身是"定投 + 移动止盈"，dca 基准是"纯定投不止盈"。对照语义 = "止盈后净效果 vs 持续定投不止盈"。dca 基准金额取每期定投金额
（用户输入）。收益口径统一：策略收益 = Σ(各次止盈落袋现金) + 期末持仓市值 - 总投入；dca 基准收益 = 期末持仓市值 - 总投入。
最大回撤按"总市值序列"算——策略总市值 = 落袋现金 + 持仓市值。口径一致，Calmar 可比。
_Avoid_: 策略自己比自己（策略含定投，dca 也含定投，口径不分会陷入"自己比自己"）

**样本外验证（Out-of-Sample Validation）**:
寻优防过拟合的机制。1 年回测窗口内按时间 0.7:0.3 切分——前 70%（train 集）网格搜索按风险调整收益选 top-k 候选（ADR-0011），后 30%（test 集）在 passed 候选中按 test Calmar 择优落库。无 passed 候选才算寻优失败，不落库报错。
_Avoid_: 前推验证（暗示滚动多窗口，本项目是单次切分）；2 年窗口（已否决，会破坏现有回测记录可比性）

**寻优落库的状态分离**:
寻优在 test 集（后 0.3 年）达标即成功，但落库走标准 calibrate（1 年全窗口回测）。两者窗口不同，可能出现"寻优成功但全窗口未通过（CALIBRATION_FAILED）"——这是职责分离的正常结果，非 bug。前端提示"寻优完成"不承诺"已通过"。
（详见 ADR-0007）

**寻优搜索维度**:
移动止盈参数均为可调优维度——启动门槛（行业 40% / 宽基 50%）、回撤比例分级表（两套，代表值行业 12% / 宽基 18%）、每次卖出比例（行业 25% / 宽基 20%）、
底仓保留（行业 25% / 宽基 40%）、卖出冷却期（20 交易日）、极端行情保护阈值（单日 7% / 连3日 12%）。按 `fundCategory` 选对应基准
出发扰动，网格搜索。
_Avoid_: 把扣款周期、每期定投金额纳入寻优（金额用户输入、周期固定对齐 dca 基准，不搜）；宽基用行业基准扰动（两套参数风险特征不同，不可混）

## 仓位管理

**无仓位上限**:
定投不设单只仓位上限、不设总仓位上限，仓位硬约束全部废弃。仓位完全由"每期定投金额 × 期数"自然累积，**止盈是唯一退出**。
用户通过"每期定投金额"自行控制仓位节奏，系统不被动减仓。
_Avoid_: 重新引入总仓位 80% / 单只 30% 硬约束（与"定投无上限"冲突，且 plannedTotalAmount 已退场无分母）

## 开发顺序

**数据源先行 / 策略主线后做**:
本期工程顺序严格串行：先把东方财富数据源完整做完（净值 + 字典 + 指数 K 线 + 自动识别 + 限流缓存 +
表级缓存），再开始策略主线（信号引擎、状态机、Service、Controller）。理由：策略主线开发时直接有真实数据可用，不用 mock；集成时无
DTO 不匹配返工；上线时没有"半手动灌入"的临时态要拆。代价是前期数小时看不到策略效果。
_Avoid_: 并行推进（B 选项，集成返工风险）；分层交付（C 选项，策略主线被切到阶段二）

## 行情数据源

**行情源（MarketData Source）**:
东方财富/天天基金公开接口。本期就接入真实实现，不走半自动灌入。三条数据线：基金净值历史（`pingzhongdata.js` 的
`Data_netWorthTrend` / `Data_ACWorthTrend`）、基金字典（`fundcode_search.js` 全量约 2 万条）、指数 K 线（
`push2his.eastmoney.com`，用于量能指标）。
_Avoid_: "本期半自动灌入"的旧定位（已升级为真实接入）

**基金子类型（fundSubType）**:
数据源维度的基金分类，区别于策略参数维度的 `FundCategory`（宽基/行业/主动/混合）。四值：`ETF`（场内交易，可直接拿自身 K 线）/
`INDEX`（指数基金，看跟踪指数）/ `INDEX_ENHANCED`（指数增强，看跟踪指数）/ `ACTIVE`（主动管理，无跟踪指数）。自动识别只走名称启发式（方法
A），未命中兜底为 ACTIVE（方法 C）；**不做持仓股票与指数成分股重合度反推**（方法 B，本期跳过，留给将来）。
_Avoid_: 把 `fundSubType` 和 `FundCategory` 合并（用途不同：前者决定数据源，后者决定止盈参数）

**跟踪/基准指数代码（benchmarkIndexCode）**:
`FundEntity` 上的可空字段。指数/ETF/指数增强基金填实际跟踪指数（如 `000300.SH`），主动/混合基金默认填沪深300 `000300.SH`。
原择时路线下用于逻辑止损判定，逻辑止损已随择时路线废弃，此字段保留供数据源识别与未来扩展用。识别流程：名称关键词命中 → 命中失败兜底为 ACTIVE（方法 A + C）。漏网基金由用户在建仓时手动补 `benchmarkIndexCode`。
_Avoid_: 主动基金强制要求填跟踪指数（主动基金本质上没有跟踪标的）

**基金类型自动识别（fundCategory 自动回填）**:
`fundCategory`（宽基/行业/主动/混合）与 `fundSubType` 一样按基金名称启发式识别，**尽力填 + 可覆盖**——识别不准时填最可能值（无关键词的指数类
默认宽基、无关键词的主动类默认主动），用户可在编辑时手动改，不留痕。不阻塞建仓流程。识别规则：指数类基金（ETF/INDEX/INDEX_ENHANCED）
名称含宽基指数词（沪深300/中证500/创业板/上证50/科创50/中证1000）→ 宽基；含行业词（半导体/医药/新能源/消费/军工/银行等）→ 行业；
两者都没命中 → 宽基。主动类基金（ACTIVE）名称含"混合/灵活配置/平衡" → 混合；否则 → 主动。
`rawName`（东方财富"稳健成长型"等风格描述）**不参与 fundCategory 判定**。
`fundCategory` 决定止盈参数（行业基金用本策略，宽基参数待定），识别准确度直接影响止盈节奏。
_Avoid_: 把识别不准的字段留 null（fundCategory 为 null 时止盈参数查不出来，阻塞后续流程）；用 rawName 判 fundCategory

**基金字典搜索（FundDict Search）**:
新建基金时用户只需输代码或名称二选一，搜索框自动补全候选列表（多候选时让用户选，A/C 份额作为不同条目并列出），选中一条后 code/name/
fundSubType/fundCategory/benchmarkIndexCode 一次性回填。字典落 `fund_dict` 表当缓存（不引入 Redis），首次查询或定时任务拉全量
`fundcode_search.js` 落库，搜索框查本地表——毫秒级响应、不撞东方财富限流、支持模糊检索。
_Avoid_: 搜索框每次按键现拉东方财富字典（撞限流）；进程内缓存（多实例不一致、重启丢失）

## 行情数据缓存

**表级缓存（MarketIndicatorSnapshot）**:
不引入 Redis，用 PostgreSQL 的 `market_indicator_snapshot` 表当缓存。每只基金每日 14:50 拉一次落库，之后所有信号生成、用户查看建议都从这张表读，不再发外部请求。若 14:50 集中拉跑不完，加 `@Scheduled` 分批触发。Redis 缓存层留给未来。
_Avoid_: 本期引入 Redis（增加基础设施复杂度）

**盘中估值（Intraday Estimate）**:
三态今日涨跌「盘中态」的数据源。来自东方财富 fundgz 接口（`fundgz.1234567.com.cn/js/{code}.js`），返回盘中估算净值（gsz）与估算涨跌幅（gszzl）。按需实时拉取，不落库——估值是短时态数据，收盘后失效。gszzl 基于单位净值，但涨跌幅是比例，同日不除权时与累计净值涨跌幅一致。拉取失败降级返 empty。
_Avoid_: 把估算净值落 fund_nav_history（那是已结算净值表，估值是短时态）

**当晚净值确认（Daily Nav Confirm）**:
让三态今日涨跌「盘后实际值」生效的机制。场外基金当日净值收盘后约 20:00 才公布。每晚 20:00-23:00 每分钟轮询所有基金，查 fund_nav_history 最近 navDate ≠ 今天 → fundgz 判 jzrq 是否 = 今天 → 是则调 pingzhongdata 拿累计净值落库。已确认跳过。用 fundgz 判定 + pingzhongdata 落库双接口，保证落累计净值而非单位净值。
_Avoid_: 用 fundgz 的 dwjz（单位净值）落库（分红基金累计净值失真）

## 盈亏与涨跌

**今日涨跌（Daily Change）**:
随交易时段演进的三态概念——盘前 = 0、盘中 = fundgz 实时估算涨跌、盘后 = 当日落库累计净值 / 昨日累计净值 - 1。判定靠"当前时间 + 当日净值是否已落库"。15:00-20:00 待公布时段显示盘中最后一次估值（带"估"标记），当晚净值落库后切换实际值。
_Avoid_: 把今日涨跌当"T-1 vs T-2"（那是昨日涨跌，旧定义已废弃）

**今日盈亏（Daily PnL）**:
随今日涨跌三态（盘前 0 / 盘中估算 / 盘后实际）。算法 = 昨日市值 × 今日涨跌幅。口径统一，不引入单位净值 gsz，规避分红基金单位/累计净值分裂。无持仓今日盈亏为 null。
_Avoid_: 用份额×(gsz-昨日单位净值) 算估算盈亏（单位净值口径，分红基金失真）

**持仓成本价（Cost Per Share）**:
每份基金的成本单价，`FundEntity` 上的维护字段，即「滚动加权平均成本」的存储形态。建仓时用户可填（不填默认 T-1 净值）；后续 INCREASE/TRANSFER_IN/INVEST 交易 CONFIRMED 时同一事务内加权更新。卖出不改单价。用于总盈亏计算与收益率/市值计算。
_Avoid_: 持仓成本总额（旧定义已废弃，改成单价）；穿透交易表实时派生（成本价是持仓属性应存储）

**总盈亏（Total PnL）**:
基金整体赚了还是亏了。盘后 = 持仓份额 ×（最近累计净值 - 成本单价）；盘中估算 = 昨日总盈亏 × (1 + 今日涨跌幅)。用于"盈利基金/亏损基金"分组。无持仓为 null。
_Avoid_: 把总盈亏和今日盈亏混为一谈（前者是累计，后者是单日）

**上涨/下跌基金 vs 盈利/亏损基金**:
两个正交维度。"上涨/下跌"按今日涨跌幅分组，"盈利/亏损"按总盈亏分组（市值 vs 成本）。一只基金可能今日上涨但整体亏损。概览页同时展示两个维度的计数。
_Avoid_: 用今日涨跌判断盈亏基金（今日涨不代表整体赚）

## 手动交易

**手动交易（Manual Transaction）**:
不经过信号、用户直接录入的交易。复用 `FundTransactionEntity`，`signalLog = null`。支持全部 5 类来源：
加仓（INCREASE）/减仓（DECREASE）/转入（TRANSFER_IN）/转出（TRANSFER_OUT）/定投（INVEST）。买入写 amount、卖出写 shares，
走 NavConfirmJob 回填另一侧。手动卖出不经过 `evaluateSignal`。入口在基金详情页"交易流水" Tab 的"手动录入"按钮。
_Avoid_: 为手动交易单独建表（复用 FundTransactionEntity 即可，signalLog=null 已是领域模型预留的手动标识）

**初始持仓录入（Existing Position Onboarding）**:
新建基金时录入已有持仓的建仓动作。`FundCreateRequest.initialMarketValue` 有值即触发——`FundStatus → HOLDING`、写 INCREASE 交易，
确认时机同步：`shares = initialMarketValue / T-1净值`，置 CONFIRMED，不等 NavConfirmJob。`initialMarketValue` 是**入仓市值**，
净值用 T-1（最近一期已公布）反算份额。`costPerShare` 可选填（不填默认 T-1 净值），>0 校验。`openedAt` 用户可填（大致建仓时点），
不填用 now，须 ≤ 今天。交易来源用 INCREASE（建仓是首笔买入非转入）。无净值历史可反算时抛 `NAV_HISTORY_EMPTY`；
openedAt 晚于今天抛 `OPENED_AT_IN_FUTURE`；`initialMarketValue` ≤ 0 或 `costPerShare` ≤ 0 抛参数校验错。
_Avoid_: 用昨日净值（语义模糊，最近一期已公布净值更准）
